> Status: approved by user 2026-05-01.
> Branch: `feat/v0.5.8-parent-admin-revamp`.
> Author: Cursor agent under the user's `/using-superpowers` workflow.
> Successor of: V0.5 follow-up (`feat/v0.5-client-followup`, merged in PR #27).

## 1. Why

After V0.5 follow-up shipped the in-device admin console, three issues surfaced in real use:

1. **Wrong orientation.** The whole app is locked landscape (battle is the design center). The admin console renders awkwardly in landscape вҖ” it's a vertical list of forms.
2. **Login friction.** The console requires JWT login on a tablet that already lives behind a parent PIN. Two passwords for the same family device is one too many.
3. **Misleading "publish new pack" button.** The button calls `POST /admin/packs/publish`, which materializes a new pack version from whatever's in the `Word` collection. But there's no UI on the device to *get words into* the collection вҖ” that path lives only on a hypothetical future web admin. So the button publishes nothing new and parents have no way to add words from the device.

V0.5.8 fixes all three:

- Lock the admin pages to portrait while they're on screen.
- Drop the JWT login UI; gate access only via the existing 6-digit parent PIN. (V0.6 will reintroduce server auth as part of multi-parent isolation.)
- Replace the "publish new pack" button with a **photo вҶ’ LLM вҶ’ review вҶ’ approve** flow that wires up the existing server-side `lessons/import` pipeline. Parents shoot a photo of a textbook page (or pick from gallery), the LLM extracts `{word, gloss_zh}` rows, the parent edits/approves, the words land in the `Word` collection. Pack publishing remains a separate one-tap on the same page.

We also rename `AdminConsole*` вҶ’ `ParentAdmin*` end-to-end (page file, route, class names, element ids, UI test ids, visible Chinese labels) to match the family-product framing.

## 3. Security tradeoff (acknowledged)

Removing `Depends(current_admin_user)` from `admin_*.py` routers makes the entire content-management surface of `https://happyword.vercel.app` reachable by any unauthenticated client.

**Threat model for V0.5.8:**
- Read leakage: anyone who knows the URL can fetch the word list, stats, and lesson drafts. The data is non-PII vocabulary; leakage is low-impact.
- Write abuse: anyone can `POST /lessons/import` (burns OpenAI credits), `POST /packs/publish` (changes what kids see), `DELETE /words/{id}` (data loss).
- Cost abuse: each `lessons/import` triggers a vision LLM call (~$0.01вҖ“0.05). A bot loop could rack up bills.

**V0.5.8 mitigations:**
- The Vercel deployment URL (`https://happyword.vercel.app`) is not publicly advertised; it sits behind security-by-obscurity.
- Vercel's per-route `maxDuration: 60` and per-account function/cost limits cap blast radius.
- The OpenAI account has a monthly soft cap.

**V0.6 plan (must land before public launch):**
- Reintroduce auth via per-family JWT tokens issued during a one-time on-device pairing flow.
- Per-family data isolation in MongoDB (every `Word`, `Category`, `LessonImportDraft` gets a `family_id` field).
- Optional rate limiting via `slowapi`.

The spec acknowledges the tradeoff explicitly; the implementation must add a banner-comment to each de-authed router pointing to this section.

## 6. Portrait orientation

### 6.1 Helper module

New file `entry/src/main/ets/utils/orientation.ets`:

```typescript
import { window } from '@kit.ArkUI';
import { common } from '@kit.AbilityKit';

/** Lock the current top window to portrait. Idempotent вҖ” if the call
 *  fails (window not ready, runtime denial), we swallow and proceed
 *  in landscape so the page is still usable. */
export async function lockPortrait(ctx: common.UIAbilityContext): Promise<void> {
  try {
    const win = await window.getLastWindow(ctx);
    await win.setPreferredOrientation(window.Orientation.PORTRAIT);
  } catch (e) {
    // log + ignore вҖ” see В§6.3 fallback behavior
  }
}

/** Restore the ability default landscape orientation. Called in
 *  `aboutToDisappear` of every page that locked portrait. */
export async function restoreAutoLandscape(ctx: common.UIAbilityContext): Promise<void> {
  try {
    const win = await window.getLastWindow(ctx);
    await win.setPreferredOrientation(window.Orientation.AUTO_ROTATION_LANDSCAPE);
  } catch (e) {
    // ignore
  }
}
```

Both functions accept `UIAbilityContext` (obtained via `this.getUIContext().getHostContext() as common.UIAbilityContext`).

### 6.2 Lifecycle hooks

ArkTS pages have four hooks we care about: `aboutToAppear` (page constructed once), `onPageShow` (page becomes visible вҖ” also fires when popping back from a pushed page), `onPageHide` (page covered by another push, but not destroyed), `aboutToDisappear` (page about to be destroyed вҖ” fires on `router.back()`).

To avoid landscape flicker during push/pop transitions inside the admin flow, only the **outer** `ParentAdminPage` restores landscape, and only when it's actually being destroyed:

`ParentAdminPage`:
- `aboutToAppear` вҶ’ `lockPortrait` (+ existing `refreshStats / refreshDrafts`)
- `onPageShow` вҶ’ `lockPortrait` (re-locks when popping back from `LessonDraftReviewPage`)
- `aboutToDisappear` вҶ’ `restoreAutoLandscape` (only fires when user backs out to ConfigPage)
- `onPageHide` вҶ’ no-op (LessonDraftReviewPage is also portrait, no restore needed during the push transition)

`LessonDraftReviewPage`:
- `aboutToAppear` вҶ’ `lockPortrait` + `client.getLessonDraft(draftId)`
- `onPageShow` вҶ’ `lockPortrait` (defensive; same-instance back is unlikely here but the call is idempotent)
- `aboutToDisappear` вҶ’ no-op (the parent `ParentAdminPage` is still portrait; restoring here would flicker landscape for one frame before `onPageShow` re-locks portrait)
- `onPageHide` вҶ’ no-op

Net result: landscape is restored exactly once, on the outermost page being destroyed, so no transition ever shows the wrong orientation.

If we ever add a third nested portrait page in the admin flow, it follows the same pattern as `LessonDraftReviewPage` (lock on appear/show, no restore anywhere except the outermost page).

### 6.3 Fallback

If orientation API fails (window not ready, OS quirk), the page still renders in landscape. UX is degraded but functional. UI tests run in an emulator that's always landscape вҖ” the test harness mocks `lockPortrait` to no-op so tests don't depend on real orientation.

### 6.4 Module manifest

`entry/src/main/module.json5` is **not changed**. The ability stays `"orientation": "auto_rotation_landscape"`. We use per-window override only.

```typescript
export interface PickedImage {

  sizeBytes: number;
}

export interface ILessonImagePicker {

  pickFromCamera(ctx: common.UIAbilityContext): Promise<PickedImage | null>;
}

export class LessonImagePicker implements ILessonImagePicker { ... }
```

Gallery: `picker.PhotoViewPicker` from `@kit.CoreFileKit`:
```typescript
const opts = new picker.PhotoSelectOptions();
opts.MIMEType = picker.PhotoViewMIMETypes.IMAGE_TYPE;
opts.maxSelectNumber = 1;
const photoPicker = new picker.PhotoViewPicker();
const result = await photoPicker.select(opts);
// result.photoUris[0] is a media:// uri вҖ” convert via @ohos.file.fs
```

Camera: launch via the system camera picker. HarmonyOS provides `cameraPicker.pick` from `@kit.CameraKit` (reference at `docs/arkts-references/api-reference.md` lines 437-467). Returns a single capture URI on success.

Both methods:
1. Call the system picker, await result.
2. On user cancel вҶ’ return `null`.
3. On success вҶ’ `fs.statSync` the returned URI to get `sizeBytes`, sniff MIME from extension (`.jpg`/`.jpeg` вҶ’ `image/jpeg`, `.png` вҶ’ `image/png`, `.webp` вҶ’ `image/webp`).
4. If MIME unsupported вҶ’ return `null` and let caller toast a friendly message.

The class accepts an injected adapter for tests:
```typescript

```

### 7.3 Multipart upload

Extend `ParentApiClient` with one new method (other lesson methods are JSON, no multipart needed):

```typescript
async importLesson(file: PickedImage): Promise<LessonDraftDto>;
```

Implementation:
1. Read file bytes via `fs.openSync(file.uri, fs.OpenMode.READ_ONLY)` + `fs.readSync` into an `ArrayBuffer`.
2. Build a multipart body manually:

```
--boundary
Content-Disposition: form-data; name="image"; filename="lesson.jpg"
Content-Type: image/jpeg

<binary>
--boundary--
```

The boundary is a random string `--happyword-${randomUuid()}`. Body assembled as `ArrayBuffer` (not string вҖ” preserves binary).

3. POST via `@kit.NetworkKit` `http.createHttp().request`:
   - URL: `${baseUrl}/api/v1/admin/lessons/import`
   - method: POST
   - extraData: the assembled buffer
   - header: `Content-Type: multipart/form-data; boundary=${boundary}`
   - read timeout: **60_000 ms** (raised вҖ” vision LLM cold start can be slow; see В§10)
   - connect timeout: 8_000 ms

4. Response handling:
   - 201 вҶ’ parse JSON into `LessonDraftDto`
   - 400/413/415 вҶ’ throw a typed `LessonImportError(code, message)` that ParentAdminPage can show inline
   - 5xx / timeout вҶ’ throw `LessonImportError('NETWORK', 'network or timeout')`

The new methods for review are plain JSON over the same `parentFetch` helper that other ParentApiClient methods use:

```typescript
async getLessonDraft(id: string): Promise<LessonDraftDto>;
async patchLessonDraft(id: string, edited: LessonExtractedDto): Promise<LessonDraftDto>;

async rejectLessonDraft(id: string): Promise<void>;
async listPendingLessonDrafts(): Promise<LessonDraftDto[]>;  // GET /lesson-drafts?status=pending
```

### 9.3 Approve flow

```
onApprove():
  busy = true;
  error = '';
  try {
    const kept = edits.filter((_, i) => keepFlags[i]);
    if (kept.length === 0) {
      error = 'иҜ·иҮіе°‘дҝқз•ҷдёҖдёӘеҚ•иҜҚ';
      busy = false;
      return;
    }
    const edited: LessonExtractedDto = {
      ...draft.extracted,
      category_label_zh: categoryLabel || undefined,
      words: kept,
    };
    await client.patchLessonDraft(draftId, edited);
    const summary = await client.approveLessonDraft(draftId);
    this.toast(`е·ІзЎ®и®Ө ${summary.inserted_word_ids.length} дёӘеҚ•иҜҚ`);
    setTimeout(() => router.back(), 800);
  } catch (e) {
    error = humanize(e);
  } finally {
    busy = false;
  }
```

## 10. Network behavior

### 10.1 Timeouts

- `importLesson`: 60_000 ms read timeout. Vision LLM + Vercel cold start can run 15вҖ“30 s; 60 s is the same ceiling as the Vercel function `maxDuration`.
- All other JSON calls: 8_000 ms (current default) is fine.

### 10.2 Loading UX

While `importBusy === true`:
- Show two distinct phases the user can see:
  - Phase 1 (вүӨ 3 s): "дёҠдј дёӯвҖҰ"
  - Phase 2 (after 3 s): "иҜҶеҲ«дёӯ, йҖҡеёё 10-30 з§’"
- A small indeterminate progress dot animation. No cancel button (multipart in flight is ugly to abort).

### 10.3 Cold-start retry

The user-facing UX does NOT auto-retry. If the first import fails with NETWORK / timeout, the page shows "дёҠдј еӨұиҙҘпјҡзҪ‘з»ңй”ҷиҜҜ [йҮҚиҜ•]". Tapping йҮҚиҜ• reuses the already-picked `PickedImage` (we keep it in state until success or explicit reset). One-button manual retry is enough вҖ” UI tests use stubbed clients and don't exercise this path.

UI test fixtures use a stub `ParentApiClient` for `LessonDraftReviewFlow.ui.test.ets`, so no real Vercel cold-start dependency.

**Documentation order (do these only AFTER implementation lands):**
- Update `docs/WordMagicGame_overall_spec.md` В§6 (12 вҶ’ 13 pages, add `LessonDraftReviewPage`), В§6.3 page-flow tree (add admin вҶ’ review branch), В§13 (server auth: "anonymous in V0.5.8, see V0.5.8 spec for V0.6 plan").
- Append V0.5.8 row to `docs/WordMagicGame_roadmap.md`.

**Risks:**
| Risk | Mitigation |
|---|---|
| HarmonyOS `cameraPicker` API differs from docs reference | Wrap behind `IPhotoPickerAdapter`; fail closed with clear error if API not available; manual tested on emulator before commit. |
| Vision LLM JSON shape drift | We don't change the server-side prompt; only consume `LessonDraftOut`. Existing `test_admin_lessons.py` already exercises the schema. |
| Open-admin abuse during V0.5.8 | Documented in В§3; not deployment-blocking for a non-public family device. |
| Per-window orientation API lacks effect on some emulators | Helper swallows; UI test stubs the call; landscape fallback is functional. |
| UI test suite renames break refs | One commit covers all renames + reference updates; CI runs full UI suite. |

**Backward compatibility:**
- AppStorage `admin_jwt`: removed read+write. Existing snapshots harmlessly contain a stale value; nothing reads it. If a future V0.6 reuses the same key it must `delete` first.
- The `LessonImportDraft.reviewer` field will contain `"parent"` for V0.5.8-era drafts; V0.6 backfill must tolerate both real usernames and the literal `"parent"`.
