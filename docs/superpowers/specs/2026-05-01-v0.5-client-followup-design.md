## 1. Why this branch exists

After `feat/v0.5.2-thru-v0.5.7` shipped on the server, the HarmonyOS client started failing on cold start with **"Failed to load words"** in `BattlePage`. Reproduces against `https://happyword.vercel.app/api/v1/packs/latest.json`, which currently returns a 1-word pack:

```json
{"version":1,"schema_version":1,"published_at":"...","words":[
  {"id":"fruit-apple","word":"apple","meaningZh":"苹果","category":"fruit","difficulty":1}
]}
```

`WordPackBootstrapper.bootstrap` reads the cache (populated by a previous background refresh that pulled this 1-word pack), parses 1 entry, hands it to a `WordRepository`, which `BattlePage.aboutToAppear` then feeds to `QuestionGenerator`. `QuestionGenerator.nextQuestion` enforces `repo.size() >= 3` and throws — the surrounding `.catch` in BattlePage swallows the throw and sets `prompt = 'Failed to load words'`. The empty option buttons + working HP/combo/timer in the live screenshot match this state exactly.

Beyond the immediate bug, the client never consumed the v0.5 server schema bumps that `feat/v0.5.2-thru-v0.5.7` shipped:

- schema_v2 — `distractors[]` / `example.{en,zh}` per word
- schema_v4 — top-level `categories[]` with server-supplied `storyZh` / `labelZh`
- schema_v5 — per-word `illustrationUrl` / `audioUrl` (Vercel Blob)

This branch closes that gap and adds an in-app, parent-PIN-gated admin shell so an operator can drive the server APIs without curl.

---

## 2. Goals / non-goals

### Goals
1. **Stop the emulator from breaking** on the next cold start.
2. **Replace the background refresh with a manual "Sync now" button** on ConfigPage. The remote pack only mutates on-disk cache when the operator asks for it; the next entry into HomePage / BattlePage adopts the freshly synced pack. Predictable for kids, friendly for UI tests, and aligned with the parent-controlled posture of the rest of the app.
3. Consume schema bumps end-to-end (v2 / v4 / v5).
4. Lazy on-disk asset cache (illustrations + audio).
5. ETag-aware client-side cache record (used by the manual sync, not by a background daemon).
6. Parent-PIN-gated admin shell inside the HarmonyOS app, debug-build entry only.
7. **Deterministic e2e test data path** (§11) — UI tests must not depend on whatever shape prod happens to be in.

### Non-goals
- New gameplay features. Question kinds, animations, coin rules unchanged.
- Server changes (server is feature-complete at V0.5.7).
- Release-build admin UI surface.
- iOS / Android / web ports of the admin shell.
- Push notifications.

## 4. Data flow

### 4.1 Cold start (no network)

```
Cold start → WordPackBootstrapper.bootstrap()
              ├── if cache present and valid (≥3 words):  use cache
              └── else:                                    use rawfile
            (no remote call, ever)
              ↓
WordRepository (+distractors/example/illustrationUrl/audioUrl, C)
              ↓                          ↘
QuestionGenerator (server distractors first)   RemoteAssetCache (D)
              ↓                                        ↓
        BattleEngine                       File URI / fallback
              ↓
   BattlePage / SpellingArea / CharacterCard
```

### 4.2 Manual sync (operator-triggered)

```
ConfigPage "同步词包" tap → WordPackSyncService.syncOnce()
                              ↓ ETag-aware GET /api/v1/packs/latest.json
                              ├── 200 → WordPackCache.write({etag, body, schemaVersion, fetchedAt})
                              ├── 304 → WordPackCache.touchFetchedAt()
                              └── err → toast, cache untouched
                              ↓
                  Toast: "已同步至 v<N>" / "已是最新 v<N>" / "同步失败 (<reason>)"
                              ↓
        Next HomePage.aboutToAppear → re-read cache → CategoryCatalog refresh + region card overlay (E)
        Next BattlePage.aboutToAppear → bootstrap reads new cache → engine starts on new pack
```

The in-memory `WordRepository` held by an active battle is **never** swapped — the operator must finish the current battle (or back out) before the new pack takes effect. This matches the existing V0.5.1 invariant and avoids word-pool swaps mid-game.

## 5. Storage shapes

### 5.1 WordPackCache v2 record (preferences key `pack_v2`)
```jsonc
{
  "etag": "\"3\"",
  "body": "<full pack JSON string>",
  "schemaVersion": 5,
  "fetchedAt": "2026-05-01T08:42:00.000Z"
}
```

`looksValid` = JSON.parse succeeds AND `body` parses to an object with `words: any[]`. Empty etag is allowed (server may not send one).

### 5.2 RemoteAssetCache on-disk index
```
${context.filesDir}/asset-cache/
  ├─ illustrations/
  │   ├─ <sha256-prefix-12>.png
  │   └─ ...
  └─ audio/
      ├─ <sha256-prefix-12>.mp3
      └─ ...
```

Index is rebuilt at process start by listing the two directories; URL→localPath is a session-scoped `Map`. LRU eviction runs when the limits are exceeded after a successful new write — never mid-render.

## 6. Error handling (cumulative across layers)

| Failure | Fallback |
|---|---|
| Cache JSON parse fails | Rawfile loader. |
| Cached pack < 3 valid words | Drop cache, rawfile loader. |
| Manual sync 304 | Keep current cache; toast "已是最新 v\<N\>". |
| Manual sync network error / non-2xx | Cache untouched; toast "同步失败 (\<code or reason\>)". |
| Asset URL download fails (timeout / 4xx / 5xx) | Bundled SVG / TTS pronunciation. Log once per URL per session. |
| Pack contains illustrationUrl / audioUrl but RemoteAssetCache hits LRU cap | Evict oldest then proceed; if still failing, fall back. |
| Admin API 401 / 403 | Toast + return to AdminLoginPage; clear stale JWT. |
| Admin API 5xx | Toast with code; stay on the current page so the operator can retry. |

## 7. Testing posture

- **Unit tests are the floor.** Every layer ships with new entry/src/test/*.test.ets cases that prove the behaviour change. Existing tests must keep passing.
- **No live network.** All HTTP calls go through `HttpRequester` (existing pattern in RemoteWordPackService); `AdminApiClient` mirrors that injection seam.
- **No live disk in unit tests.** `RemoteAssetCache` accepts an injectable filesystem-like adapter the same way `WordPackCache` accepts `StringPreferencesLike`.
- **UiTest smoke for admin pages.** Each admin page gets a render-and-bind test that asserts the page composes without throwing and a key element (login form, list row, upload button) is visible. Full e2e is deferred to V0.6.

## 11. E2E test data determinism

The HarmonyOS UiTest suite (`entry/src/ohosTest`) currently runs against whatever the live `https://happyword.vercel.app` happens to serve. Right now that's a 1-word pack, tomorrow it could be 50 words with full v5 metadata, the day after it could be a half-published pack mid-rollout. Tests cannot make stable asserts on top of that.

This branch fixes the underlying drift in three layers — they stack, so a test author picks the lowest one that still gives them what they need:

### 11.1 Layer 1 — Rawfile is the deterministic floor

After VC0.5.A lands, **bootstrap never touches the network on cold start**. With a fresh `pack_v2` preferences slot, every cold start uses the bundled `data/words_v1.json` (50 words, version-controlled, never drifts). UiTest scenarios that don't care about server-driven content just:

1. Wipe the test ability's preferences before the run (existing `WordPackCache.clearForTest(ctx)` will be added in VC0.5.A so tests can call it without touching internals).
2. Launch the app.
3. Assert against the rawfile's known 50-word universe. The rawfile is git-tracked, so its contents are part of the test contract.

This covers ~80% of the existing UiTest cases — battle gameplay, monster catalog, today plan, coin flow.

### 11.2 Layer 2 — Sync-from-fixture for tests that DO care about server-driven content

Tests that exercise the manual sync flow, schema_v2/v4/v5 features, or HomePage storyZh overlay need a deterministic remote response. The **test runner** controls the URL via a build-mode-gated override:

- New `RemoteWordPackConfig.overrideLatestPackUrl(url: string | undefined)` (clears on undefined). Only honoured in `BUILD_MODE_NAME === 'debug'`.
- Tests set the override to a bundled fixture URL: `resource://rawfile/data/test_pack_fixture_v5.json` — a new fixture rawfile we ship under `entry/src/ohosTest/resources/rawfile/data/test_pack_fixture_v5.json`. The fixture covers all schema_v5 features (3 categories with storyZh, 9 words including distractors, example, illustrationUrl pointing to a bundled SVG, audioUrl pointing to a bundled OGG).
- `RemoteWordPackService.fetchLatest` learns to read `resource://rawfile/...` URIs by routing them through `resourceManager.getRawFileContent` instead of httpRequest. Path is debug-only; release builds reject any non-`https://` URL with the existing error path.
- Tests then drive the manual sync flow against this fixture, assert exact pack content, and clean up by clearing the override.

This is the recommended path for any UI test that cares about server-driven content — it gives a stable JSON without spinning up a real server.

### 11.3 Layer 3 — Live-prod end-to-end (opt-in only)

Some smoke tests genuinely want to verify the full HTTP round-trip against `happyword.vercel.app`. Those tests:

1. Are tagged `@LiveProd` and skipped by default (mirrors the server's `LIVE_OPENAI=1` opt-in pattern).
2. Run AFTER an operator (or CI step) seeds prod into a known state via the admin scripts: `seed_from_rawfile.py` → `publish_pack.py`. This guarantees prod has the 50 rawfile words at a known version.
3. Assert at the structural level only (`schema_version >= 1`, `words.length >= 50`, `words[0].id` is a valid kebab-case slug). They do NOT pin specific word values, because the seed contents may diverge from the rawfile in the future.

CI does not run `@LiveProd` tests on every commit. They're a release smoke gate, run manually before tagging.

### 11.4 Cache reset helper

VC0.5.A ships `WordPackCache.clearForTest(ctx)` as a small public helper that wipes the `pack_v2` preferences slot. UiTest setup uses it; production code does not call it. Documented in the test conventions section of `docs/server/operations.md` (added in VC0.5.B since that's when ConfigPage shipped).
