# V0.6 家长账户与设备绑定版 — 总体设计

- **Date:** 2026-05-01
- **Status:** Design-for-implementation; revision 4 — 邮件主路径切到 Gmail SMTP（aiosmtplib + App Password），引入 `EmailProvider` 抽象
- **Roadmap:** [WordMagicGame_roadmap.md §13 V0.6](../../WordMagicGame_roadmap.md)
- **Target sub-version range:** V0.6.1 → V0.6.7（拆 7 个 walking-skeleton 子版本）
- **Tech stack additions:** FastAPI + Jinja2Templates + HTMX + Tailwind CDN（家长 web）；`qrcode[pil]`（家长 web 端 QR 渲染）；`aiosmtplib`（异步 SMTP 客户端，Gmail App Password 主路径，OTP + 通知共用）+ `EmailProvider` 接口预留 Resend / SES 切换口；HarmonyOS 端启用 `@kit.ScanKit` + `@kit.AssetStoreKit` 两个系统能力，无新增 oh-package 依赖
- **Out-of-scope:** 句子填词 / AI 剧情（V0.7）、BattleAudioMixer（V0.8）、Cocos2D 重写（V0.9）、真实零钱/支付宝兑换（无限期延后，独立合规设计）

## Revision history

- **r4 (2026-05-01):** 邮件主路径从 Resend 切到 **Gmail SMTP（aiosmtplib + App Password）**；新增 `EmailProvider` 协议（默认 `GmailSmtpProvider`，Resend / SES 留作后续切换），`notification_service` 调接口而非具体实现；环境变量 `RESEND_*` 全部替换为 `SMTP_*`；§12 风险表新增 Gmail 500/日 上限 + From 必须等于登录账号的口径。
- **r3 (2026-05-01):** 家庭词包模型从「单 family → 单 versioned pack」改为「单 family → **多个独立 pack**，每包 ≤ 50 词，可无限新建（建议每个课本单元一个）」；家长账号注销 grace 由 72h 调到 **7 天**；OTP 邮件**仅中文**；cookie 续期阈值确定为 **iat > 7 天**；家庭词包 publish 后客户端**不推送同步**，UI 仅提示「家长已更新词包，请到设置→同步词包」。
- **r2 (2026-05-01):** 三处颠覆性调整 + 一项新核心能力：(a) 家长账号注册改为「邮箱 OTP 验证码 + 无密码」magic-code 模型；(b) 配对方向反转 — 家长 web 渲染 QR，客户端用 ScanKit 扫码；(c) device_id 来源改用 `@kit.AssetStoreKit` 持久化 UUID 实现「卸载重装不变」；(d) 新增 V0.6.3「家庭私有词包」子版本，绑定设备拉家长定制的词包。
- **r1 (2026-05-01):** 初版（密码注册 / 设备生成配对码 / preferences UUID / 家长账号 → 多孩子档案多对一）。

## 1. 背景与一句话设计

### 1.1 问题背景

V0.5.7 完成后，服务端已经具备「内容生产 + 版本化发布 + 客户端 ETag 缓存」的完整链路，单设备体验闭环也已抛光（V0.4.8）。但产品仍卡在两个根本问题上：

1. **学习数据是死锁的**。`LearningRecorder` / `WishlistStore` / `CoinAccount` 全部基于 `@ohos.data.preferences`，换平板就归零；家长无法看到孩子学了什么、薄弱在哪。
2. **奖励与内容闭环只在客户端完成**。V0.3.9 的 PIN 弹窗 + GiftBox 把「家长确认」抽象成本地一段代码；admin 词库是全局的，家长无法基于自己孩子的进度定制内容。

V0.6 的目标是在不破坏 V0.5.7「离线优先」契约的前提下，把这两条链路打开：让家长有一条独立的 web 端入口，能看到孩子的学习画面、能配置/审批兑换、能定制孩子专属的词包；让设备从「孤岛」升级为「绑定到一个家庭、且持有一个孩子档案的视图」。

### 1.2 一句话设计

> 家长用邮箱 OTP 验证码注册 / 登录（无密码）；家长 web 渲染绑定二维码，客户端用 `@kit.ScanKit` 扫码完成绑定；设备身份用 `@kit.AssetStoreKit` 持久化 UUID（卸载重装不变）；每个绑定的设备自动获得一个 1:1 的孩子档案；学习记录与兑换记录按 child_profile_id LWW 同步；新增「家庭词包」让家长可为自己孩子按课本单元发布**多个**独立词包（每包 ≤ 50 词），客户端把所有 active 家庭词包合并 overlay 在全局词包之上；客户端未绑定时行为完全保留 V0.5.7。

### 1.3 成功标准

- V0.6.1 完成时：家长输邮箱收 OTP，输入验证码后落地家长账号 + Family，进入空 dashboard。
- V0.6.7 完成时：一个家长可绑定 ≥1 台 HarmonyOS 设备；每台设备有独立孩子档案；家长能在 web 编辑家庭词包并 publish，绑定设备能拉到这份词包；学习记录、愿望单、兑换记录在设备 ↔ 云端 LWW 同步；家长能在 web 端审批兑换并收到邮件通知；家长能注销账号并联级清理云端数据。
- HarmonyOS 客户端 38+ 单元 / UI 套件 0 回归；离线冷启动仍能进入战斗（rawfile + cache 兜底契约不破）。
- 服务端 `uv run pytest` 0 errors / 0 warnings 不松动；家长 web 路由通过 FastAPI TestClient + BeautifulSoup 做 HTML 结构断言。

## 2. 关键决策记录

| 维度 | 选择 | 理由 |
| --- | --- | --- |
| 家长账号模型 | **邮箱 OTP 验证码注册 + 登录（无密码）**；session cookie 30 天滚动续期 | 家长是低频访问，密码记不住反而是负担；OTP 顺带完成邮箱所有权验证；通知能力（V0.6.7）共享同一邮件链路，无需第二条 |
| 家长端形态 | FastAPI + Jinja2Templates + HTMX + Tailwind CDN，与 API 同 Vercel 项目 | 单人开发，零新构建管线；pytest+TestClient 直测 HTML；后续 V0.7 可渐进迁移到 Vite + React |
| 配对方向 | **家长 web 渲染 QR，客户端用 `@kit.ScanKit` 扫码**；同一页面同时显示 6 位短码兜底（手动输入） | 与现实直觉一致（家长打开 web、孩子拿设备扫）；ScanKit 是系统能力，调用原生 UI 体验最稳；QR 内容是 HTTPS URL，被任意 QR 阅读器读到也只会落到「请用 WordMagic app 扫码」引导页 |
| 客户端 device_id | **`@kit.AssetStoreKit` 持久化随机 UUID（IS_PERSISTENT=true）**；不可用时降级到 preferences UUID（卸载重装会变） | ODID 在「同一开发者所有 app 都卸载」时会重新生成（happyword 是单 app，等价于卸载重装即换号）；AssetStoreKit `IS_PERSISTENT: true` 是 HarmonyOS NEXT 唯一原生「卸载重装仍可读回」机制；不引入华为账号绑定 |
| 数据所有权 | 设备本地为真，云端为镜（LWW by `lastAnsweredMs`） | 保留 V0.5.7「离线可用」契约；多设备同孩子是次要场景；冲突解决最简 |
| 兑换流程 | 未绑定设备走 V0.3.9 PIN 即扣；已绑定设备走「申请 → 家长 web 确认 → 设备 poll → 扣币 + GiftBox」 | V0.3.9 资产（PIN dialog / GiftBox 动画）100% 复用；离线仍可玩；家长真正掌握审批权 |
| 家庭词包 | **多包模型**：family_pack_definitions（pack 定义，多条/family）+ family_word_packs（per-pack versioned 快照）+ family_pack_pointers（per-pack 当前/前一版本）+ family_pack_drafts（per-pack 草稿）四张表；**每包 ≤ 50 词**；客户端拉单端点拿到该 family 全部 active packs 合并 JSON | 路线图的「家长可以定制内容」从 V0.6 起兑现；按课本单元分包匹配实际教学节奏；50 词上限让家长聚焦每包内容、单包 publish 性能稳定；客户端始终是单 ETag 缓存，不爆缓存 slot 数 |
| 鉴权 | 复用 V0.5.1 `User` + JWT；role=parent 的用户走 cookie session（家长 web）；设备走独立的 device-token JWT | 单一 User 表多 role 模型；middleware 按 sub 前缀分流（`parent:<id>` / `device:<binding_id>` / `admin:<u>`） |
| 设备 - 孩子档案 模型 | **每台绑定的设备 1:1 持有一个 ChildProfile**（绑定瞬间自动创建空档案）；一个家长可绑定多台设备 | 完全匹配「家庭里每个孩子配一台平板」的真实场景；UI 简洁；后续多家长 / 多孩子共享一台设备的复杂场景留给 V0.7+ |
| 通知 | V0.6.7：邮件（Gmail SMTP via `EmailProvider`）+ 家长 web inbox 主动 poll；不接 HarmonyOS Push Kit | 真 push 需平台资质 + 后端常驻服务；邮件 + poll 已能覆盖「孩子兑换 → 家长审批」时延期望（≤几分钟） |
| 同步频率 | app `onBackground` / 战斗 ResultPage 出场 / ConfigPage「立即同步」按钮触发；不做后台定时同步 | V0.5.B「同步词包」按钮已经训练用户接受手动模型；省电；与家长审批的时延语义一致 |
| 家长 web 路由前缀 | 页面在 `/parent/*`，API 复用 `/api/v1/parent/*`，公开 `/api/v1/pair/*` | 与现有 `/api/v1/admin/*` 并列，单 FastAPI 实例 |
| 静态资源 | Tailwind 走 CDN；HTMX 走 CDN；自家 CSS/JS 极少，落 `server/static/` | 不引 npm；不引 build；锁定 Tailwind play CDN 仅用于内部工具页 |
| 邮件 | **Gmail SMTP（`smtp.gmail.com:587` STARTTLS + App Password）via `aiosmtplib`** 作为 V0.6.1 默认实现；`app/services/email_provider.py` 定义 `EmailProvider` Protocol（`async def send(*, to, subject, html, text) -> None`），默认注入 `GmailSmtpProvider`；`notification_service.send_email/send_otp_email/send_*_notification` 全部经 provider 间接调用，未来切 Resend / SES 只动一个文件 | Gmail 免费账号 500 封/日（Workspace 2000）足够单家庭月级使用；零运营成本；From 必须 == 登录邮箱（§12 风险表已记）；接口隔离避免重写业务代码 |
| 拆分策略 | Walking Skeleton（先打通 e2e，再加深），同 V0.5 | 基础设施风险前置；任意子版本可上线 |

**关键约束：**

- 单 Vercel 项目，单 FastAPI 实例。家长 web 是 FastAPI 的一组 HTML route，不是独立部署。
- HarmonyOS 客户端永远不直接渲染家长 UI；家长操作全部在 web 端完成。
- 客户端冷启动路径 V0.5.7 已稳定，本版只**追加**「sync 入口」+「绑定设备时 family pack 拉取」，不改动 cache / rawfile fallback 顺序。
- 设备 token 与家长 cookie 是不同 sub claim 的 JWT；服务端中间件按 `sub` 前缀分流（`parent:<id>` / `device:<binding_id>` / `admin:<username>`）。
- 邮件链路从 V0.6.1 起就是必备依赖（OTP）；从 V0.6.7 起复用为通知通道。

## 5. 数据模型

新集合按子版本顺序声明（旧集合 V0.5 已经定义，本版不改 schema）。

### 5.1 `users` 字段扩充（V0.6.1 起）

```python
class UserRole(StrEnum):
    ADMIN = "admin"
    PARENT = "parent"

class User(Document):
    username: Annotated[str, Indexed(unique=True)]
    password_hash: str | None = None              # CHANGE V0.6.1: parent 用户为 None；admin 仍是 bcrypt
    role: UserRole = UserRole.ADMIN
    family_id: str | None = None                  # NEW V0.6.1, parent 必填，admin 必空
    display_name: str | None = None               # NEW V0.6.1，家长 web 头像名（首次注册可缺，UI 用 email 前缀兜底）
    email: str | None = None                      # NEW V0.6.1，parent 必填且 == username；admin 可空
    timezone: str = "Asia/Shanghai"               # NEW V0.6.1，邮件落款 / 周报触发时区
    created_at: datetime
    last_login_at: datetime | None = None
    scheduled_deletion_at: datetime | None = None # NEW V0.6.7，注销 7d grace
```

`username` / `email` 对 `role=parent` 必为合法 email；admin 沿用 V0.5 短账户名。校验在 schema 层。

### 5.2 `email_verifications`（V0.6.1 起）

```python
class EmailVerification(Document):
    email: Annotated[str, Indexed()]
    code_hash: str                                # bcrypt hash of 6-digit code（不存明文）
    purpose: Literal["signup_or_login"] = "signup_or_login"  # 留出未来 "delete_confirm" 等
    attempts: int = 0                             # 校验失败次数；≥5 直接 expired
    status: Literal["pending", "used", "expired"]
    requesting_ip: str | None = None              # 仅诊断；GDPR 30 天后清
    created_at: datetime
    expires_at: datetime                          # created_at + 10 min

    class Settings:
        name = "email_verifications"
        indexes = [[("email", 1), ("status", 1), ("expires_at", 1)]]
```

服务端清理任务（V0.6.7）每小时把 `status=pending AND expires_at < now` 的 doc 标 `expired`。

### 5.3 `families`（V0.6.1 起）

```python
class Family(Document):
    id: Annotated[str, Indexed(unique=True)]      # 业务 id, slug 形式 fam-<8-hex>
    primary_parent_id: str                        # users._id of the seat-zero parent
    display_name: str | None                      # 例 "汪汪家"，可空
    created_at: datetime
    updated_at: datetime

    class Settings:
        name = "families"
```

V0.6 仅支持单家长，`primary_parent_id` 等价于「这个 family 的家长」；V0.7+ 可加 `parents: list[str]` 字段做多家长。

### 5.4 `pair_tokens`（V0.6.2 起）

```python
class PairToken(Document):
    token: Annotated[str, Indexed(unique=True)]   # 32 字节随机 base64url（QR payload 主体）
    short_code: Annotated[str, Indexed(unique=True)]  # 6 位 base32 (避混淆 0/O/1/I)，手输兜底
    family_id: Annotated[str, Indexed()]
    parent_user_id: str                           # 哪个家长发起的（多家长时区分，目前等价于 family.primary_parent_id）
    status: Literal["pending", "redeemed", "expired", "cancelled"]
    binding_id: str | None = None                 # redeem 后回填
    device_id: str | None = None                  # redeem 后回填
    created_at: datetime
    expires_at: datetime                          # created_at + 3 min
    redeemed_at: datetime | None = None

    class Settings:
        name = "pair_tokens"
        indexes = [[("status", 1), ("expires_at", 1)]]
```

`token` 是 QR 内容的核心；`short_code` 是手输兜底。两者绑定同一行 doc，redeem 时任意一个都可以。

### 5.5 `device_bindings`（V0.6.2 起）

```python
class DeviceBinding(Document):
    id: Annotated[str, Indexed(unique=True)]      # 业务 id, slug 形式 bind-<8-hex>
    device_id: Annotated[str, Indexed()]          # 客户端 AssetStoreKit 持久化 UUID（不变跨重装）
    family_id: Annotated[str, Indexed()]
    paired_at: datetime
    revoked_at: datetime | None = None            # 家长 web 解绑、孩子档案被删、家长账号注销
    last_seen_at: datetime | None = None
    user_agent: str | None = None                 # 客户端 redeem 时上报，仅诊断

    class Settings:
        name = "device_bindings"
        indexes = [[("device_id", 1), ("family_id", 1)]]
```

> 同一 `device_id` 在某个 family 下**至多一条 active 绑定**（revoked_at IS NULL）；服务端 redeem 时如果检测到旧 active 行，先 set `revoked_at = now()`，再插入新行。这覆盖「同一设备先绑定 family A，再扫描 family B 的 QR」的现实场景。

### 5.6 `child_profiles`（V0.6.2 起，1:1 with binding）

```python
class ChildProfile(Document):
    id: Annotated[str, Indexed(unique=True)]      # 业务 id, slug 形式 child-<8-hex>
```

### 5.7 家庭词包四张表（V0.6.3 起）

家庭词包采用「pack 定义 + 草稿 + 版本快照 + 指针」四张表的多包模型。一个 family 可以同时拥有任意多个独立 pack（建议每个课本单元一个）；每个 pack 单独维护 draft / 发布版本 / 指针；客户端拉到的是「该 family 所有 active pack 合并出的单 JSON」，但服务端按 pack 各自管理。

```python
class FamilyPackDefinition(Document):
    """一个 family 内的家庭词包元数据。多条 / family。"""
    id: Annotated[str, Indexed(unique=True)]      # 业务 id, slug 形式 pck-<8-hex>
    family_id: Annotated[str, Indexed()]
    name: str                                      # 1..32 字符，例 "三年级第三单元"
    description: str | None = None                 # 0..200 字符，可空

    updated_at: datetime
    archived_at: datetime | None = None
    created_by_parent_id: str

        name = "family_pack_definitions"
        indexes = [[("family_id", 1), ("state", 1), ("updated_at", -1)]]


class FamilyPackDraft(Document):
    """每个 pack definition 1:1 对应的草稿区。publish 时序列化为 FamilyWordPack。"""
    pack_definition_id: Annotated[str, Indexed(unique=True)]   # 1:1 映射
    family_id: Annotated[str, Indexed()]
    words: list[dict]                              # ≤ FAMILY_PACK_MAX_WORDS = 50
    updated_at: datetime
    updated_by_parent_id: str

    class Settings:
        name = "family_pack_drafts"


class FamilyWordPack(Document):
    """已发布的版本快照。每个 pack definition 多条版本。"""
    pack_definition_id: Annotated[str, Indexed()]
    family_id: Annotated[str, Indexed()]
    version: int                                   # 单调递增，per pack_definition 独立
    words: list[dict]                              # ≤ 50；与全局 WordPack.words 字段同形
    schema_version: int                            # 1 / 2 / 4 / 5（与全局一致）
    published_at: datetime
    published_by_parent_id: str
    notes: str | None

    class Settings:
        name = "family_word_packs"
        indexes = [[("pack_definition_id", 1), ("version", -1)]]

    """每个 pack definition 1:1 的当前指针（current/previous）。"""
    pack_definition_id: Annotated[str, Indexed(unique=True)]
    family_id: Annotated[str, Indexed()]
    current_version: int
    previous_version: int | None
    updated_at: datetime

        name = "family_pack_pointers"
```

**约束与不变量：**

- `FAMILY_PACK_MAX_WORDS = 50`（const，写在 `app/config.py`）；publish 时 hard reject 超 50 词。
- `FamilyPackDefinition.name` 在同一 family 内唯一（数据库 unique compound index `(family_id, name)` where `state=active`）；归档后名字可被新 pack 复用。
- 一个 family 没发布过任何 pack 时，`/api/v1/child/family-packs/latest.json` 返回 204；客户端只走全局 pack。
- 单个 pack 没发布过任何版本（只有 draft）→ 不出现在客户端合并 JSON 里。
- `FamilyPackDefinition.state == "archived"` 的 pack **不参与**客户端合并；归档是「家长想暂停某课本单元但不删数据」的语义。
- 删除（DELETE definition）= state=archived（软删）；硬删要走 V0.6.7 的账号注销联级。

**家长 web 编辑器里词条来源（沿用 r2 的 4 类，scope 改为 per-pack）：**

1. **从全局 admin 库挑词**（按 category / difficulty 过滤选）— 多数情况；
2. **自己写**（提交单条 `{ id, word, meaningZh, category, difficulty }`，id 必须以 `fam-<family_id_8>-` 前缀，避免与全局重名）；
3. **从全局库继承再编辑**（克隆全局词，修改 meaningZh / 难度，本家庭专属覆盖）；
4. **删除全局某词在本家庭的可见性**（本质是把该词加入某 pack 并标 `hidden: true`，客户端合并时识别为「不要这个词」）。

**跨 pack 同 word_id 的合并语义（服务端打包 + 客户端 mergeRepos 共享同一规则）：**

- 同一个 `word_id` 出现在多个 active pack 里时，按 `published_at` 最新的 pack 版本胜出（行为：parent 在 pack-A publish 之后又在 pack-B publish 同一个词的更新版，pack-B 胜出）。
- 任一 active pack 里出现 `{ id: <gid>, hidden: true }` → 客户端最终结果排除 `<gid>`，**即使其他 active pack 里也含同 gid 的非 hidden 版本**（hidden 是「在本家庭明确不要」，强于 override）。
- 跨 pack 重复 word_id 在家长 web 端**warning 不阻止**，列表上显示 `⚠ 该词与其他词包重复，发布后只会保留最新版本`。
- 服务端 publish 时不做跨 pack 校验（per-pack 独立 publish）；冲突解析全部在打包 JSON 时执行。

```jsonc
{
  "schema_version": 5,
  "family_id": "fam-...",
  "merged_at": "2026-05-12T03:14:15Z",
  "packs": [
    { "pack_id": "pck-...", "name": "三年级第三单元", "version": 2,
```

`ETag` = `"sha256({pack_id}:{version} 顺序排序拼接)"`，客户端 `If-None-Match` 命中时 304 不返回 body。  
该端点服务端保证：
- `packs[]` 仅含 `state=active` 且 `pointer.current_version != null` 的 pack。
- `packs[]` 内每个 pack 自身的 `words[]` 是该 pack 当前 published 版本快照（不再做跨 pack 合并 / 去重 — 合并语义在客户端 mergeRepos 里执行，§7.6）。
- 顶层 `schema_version` = `max(packs[*].schema_version)`，方便老客户端按版本 gate 字段使用。

## 7. 协议与同步细节

### 7.1 鉴权拓扑

```text
JWT sub 前缀  | 角色          | 颁发时机                              | 主要消费 Path                  | 载体
-----------  | -----------   | -------------                         | -----------                    | ----
admin:<u>    | admin         | /api/v1/auth/login（V0.5 已有）        | /api/v1/admin/*                | Authorization: Bearer
parent:<id>  | parent        | /api/v1/parent/auth/verify-code（V0.6.1）| /api/v1/parent/* + /parent/*   | Cookie wm_session（HttpOnly + SameSite=Lax + Secure，30 天滚动续期）
device:<id>  | bound device  | /api/v1/pair/redeem（V0.6.2）          | /api/v1/child/* + /api/v1/pair/status | Authorization: Bearer
```

中间件按 sub 前缀分流；任意端点显式声明可接受集合。误用一律 403 `FORBIDDEN`。

家长 cookie 续期：每次成功的 cookie-auth 请求若距 `iat > 7 天`，服务端重发新 cookie；30 天上限触发后引导重新走 OTP。

```text
   │ code { email }               │                            │
   ├─────────────────────────────►│                            │
   │                              │ generate 6-digit code       │
   │                              │ insert EmailVerification    │
   │                              │ send email via EmailProvider│
   │                              ├───────────────────────────►│
   │ ◄── 202 { expires_in: 600 }  │                            │
   │                              │                            │
   │                              │                  email arrives,
   │                              │                  user reads code 123456
   │                              │                            │
   │ POST /parent/auth/verify-    │                            │
   │ code { email, code }         │                            │
   ├─────────────────────────────►│                            │
   │                              │ bcrypt verify code          │
   │                              │ if user not exists:         │
   │                              │   create Family + User      │
   │                              │ issue 30d cookie JWT        │
   │                              │ EmailVerification.status=used│
   │ ◄── 200 + Set-Cookie         │                            │
   │      { family_id, parent }   │                            │
```

**速率与防滥用：**

- 同 email 1 分钟内只接受 1 次 `/request-code`（数据库内行锁 by email）；超出返回 202 但跳过实际邮件发送（不暴露 email 是否已存在）。
- 同 email 10 分钟内 ≥ 3 次失败的 verify → 把当前 active EmailVerification 标 expired，要求重发码。
- 单 EmailVerification 最多 5 次 verify 尝试，溢出 410 `TOO_MANY_ATTEMPTS`。
- email 枚举防御：`/request-code` 永远 202，攻击者无法用响应区分邮箱是否已注册。

**QR payload 形状：**

```text
https://happyword.vercel.app/p/{first-12-chars-of-token}
```

只放 token 前 12 字符（短，QR 码密度低）；客户端扫到后对照 ScanKit 返回的字符串前缀解析，再调 `/pair/redeem` 时把完整路径段当 `token` 提交（服务端按 `{token startswith request_token}` 找 pending 行）。这避免 QR 内容超过中等密度二维码极限（≈100 字符）。短码 6 位 base32 是另一条路径。

**为什么用 HTTPS URL 而非 `wm-pair://...` 自定义 scheme：**

- 兼容任意 QR 阅读器；用户用其他 app（华为相机 / 微信扫一扫）扫到 QR 会落到 `/p/{tag}` 页面，看到「请用 WordMagic app 扫码绑定」+ 6 位短码兜底。
- 服务端 `/p/{tag}` GET 路由本身**不**完成 redeem，只是引导页 — redeem 必须由客户端调 `/api/v1/pair/redeem`。中间人看到 QR 也无法仅凭它绑定设备。

**取消 / 过期：**

- 家长 web 上「取消」按钮 DELETE `/parent/pair/{token}` → status=cancelled。
- 3 分钟未 redeem → status=expired，HTMX 状态更新为「已过期，重新生成」按钮。
- 客户端 ScanBindingPage 扫到已过期 token → /redeem 返回 410，UI 提示「二维码已过期，请家长重新生成」+ 重新扫码按钮。

### 7.4 客户端 device_id 持久化（V0.6.2）

```text
DeviceIdProvider.get():
    1. Try AssetStoreKit:
         asset = Asset.query({ alias: "wordmagic.device_id" })
         if asset exists → return asset.value
    2. Generate fresh UUID:
         id = randomBytes(16).toHex()        # 32 hex chars
    3. Store via AssetStoreKit:
         Asset.add({ alias: "wordmagic.device_id",
                     value: id,
                     accessibility: DEVICE_FIRST_UNLOCKED,
                     is_persistent: true })
       on failure (e.g. permission denied / API not available):
         fall back to preferences "wordmagic_device" key "device_id"
         emit log event "device_id.fallback_to_prefs"
    4. Return id.
```

权限：`module.json5` 加 `ohos.permission.STORE_PERSISTENT_DATA`（ASSET 持久化必需）。首次启动 EntryAbility.onCreate 触发一次 `DeviceIdProvider.get()`；ID 在 app 整个运行周期内是单例缓存。

UI 提示：ConfigPage「家长账户」组在「绑定家长」入口下方加一行小字 `🔒 设备 ID 由系统持久化保存（卸载重装不变）`；如果检测到走了 fallback 路径，文案改为 `⚠️ 设备 ID 仅本机有效（卸载重装会重新绑定）` 并附 1 行「为何如此」帮助链接（指向 `/parent/help/device-id`）。

### 7.5 学习记录 LWW 同步（V0.6.4）

**触发点（客户端）：**

1. `BattlePage` `aboutToDisappear`（一局结束、回 ResultPage 之前）。
2. `EntryAbility.onBackground`（app 切到后台）。
3. `ConfigPage` 的「立即同步学习记录」手动按钮。

每次触发跑一次 `CloudSyncService.syncOnce()`：

```text
1. 读 LearningRecorder.statsForTest()，过滤 lastAnsweredMs > localCheckpointMs
2. 分批 push（每批 ≤200 项）到 POST /api/v1/child/word-stats/sync
3. 服务端按 (child_profile_id, wordId) 合并（LWW by lastAnsweredMs）
4. 客户端拿到 server_pulls：
     - 本地 stat.lastAnsweredMs < pull.lastAnsweredMs → 覆盖本地 stat
     - 否则忽略
5. 客户端把成功 push 的 max(lastAnsweredMs) 写入 localCheckpointMs
6. flushNow LearningRecorder
```

**关键不变量：**

- 本地 LearningRecorder 仍是单设备真理。
- 多设备同孩子：晚答题覆盖早答题；counter 字段会丢中间增量但聚合方向单调。
- 网络失败：syncOnce 静默失败，下次再试；不阻塞 ResultPage 渲染。

### 7.6 家庭词包合并（V0.6.3，多包模型）

**客户端 cold-start（绑定状态下）：**

```text
WordPackBootstrapper.bootstrap():
  global_pack = WordPackCache.read("pack_v2")              # V0.5.7 标准 cache
  if global_pack and >=3 valid words:
      base_repo = WordRepository.fromPackJson(global_pack)
  else:
      base_repo = WordRepository.fromRawfile()

  if not device.bound:
      return base_repo                                      # V0.5.7 unchanged

  family_packs_blob = WordPackCache.read("family_packs_v1")  # V0.6.3 新 slot
  if family_packs_blob is null or family_packs_blob.packs is empty:
      return base_repo

  return mergeFamilyPacks(base=base_repo, overlay=family_packs_blob)
```

**`mergeFamilyPacks(base, blob)` 的合并规则（cumulative）：**

```text
1. 收集所有 hidden id：
     hidden_ids = { entry.id for pack in blob.packs for entry in pack.words if entry.hidden }
2. 收集所有 non-hidden 词条，按 (publish_time desc, pack_id asc) 排序后建 word_id → entry 字典：
     latest_overlay = {}
     for pack in sorted(blob.packs, key=(pack.published_at desc, pack.pack_id asc)):
         for entry in pack.words:
             if entry.hidden: continue
             if entry.id not in latest_overlay:
                 latest_overlay[entry.id] = entry             # 最新 publish 胜出
3. 输出 repo：
     out = []
     for global_entry in base:
         if global_entry.id in hidden_ids: continue
         if global_entry.id in latest_overlay:
             out.push(latest_overlay[global_entry.id])        # 同 id 用 overlay 版本
         else:
             out.push(global_entry)
     for overlay_id, overlay_entry in latest_overlay:
         if overlay_id not in (base.ids ∪ hidden_ids):
             out.push(overlay_entry)                          # family 独有词加入
     return WordRepository.fromEntries(out)
```

**关键不变量：**

- hidden 是「家长在本家庭明确不要这个词」，**强于** override 与 add。任一 active pack 标 hidden 即整个家庭排除该词。
- 同 id 跨 pack 重复时按 published_at desc 解析；客户端不依赖服务端做去重（服务端只负责按 pack 切片）。
- `fam-<family_id_8>-` 前缀的自定义词只可能出现在某一个 pack 中（id 全局唯一约束在服务端层）；不可能与全局重名。
- 全局独有 + 不被任何 pack 隐藏 + 不被任何 pack override 的词 → 原样保留（即家长建了 5 个家庭包并不会让全局 50 词消失）。

**手动同步（V0.5.B 按钮复用）：**

ConfigPage「同步词包」按钮原本只拉全局；V0.6.3 起，绑定设备点这个按钮会顺序：

1. `GET /api/v1/packs/latest.json` with ETag → 304/200 → 写 cache `pack_v2`。
2. `GET /api/v1/child/family-packs/latest.json` with device_token + ETag → 200/304/204 → 写 cache `family_packs_v1` 或清空该 cache。
3. Toast 文案累加：例如「全局已是最新 v3 / 家庭包已同步至 v2 v1（共 2 包 78 词）」。

**publish-后通知策略（r3 决议）：**

- 家长在 web publish 一个 pack 后，**服务端不向客户端推任何信号**。
- 已绑定客户端的 HomePage `aboutToAppear`（每次回到首页）执行一次轻量 `HEAD /api/v1/child/family-packs/latest.json`（仅读 ETag header），若 ETag 与本地 cache 不一致 → 显示一条非阻塞 banner：「家长已更新词包，请到设置 → 同步词包」，不强拉、不替换 in-memory repo。
- banner 在用户主动到 ConfigPage 触发同步后或 24 小时未点击后自动隐藏。
- 此设计保留了 V0.5.B 的「家长控制 + 手动触发」语义，避免战斗中词池突变。

**HEAD 请求的失败容忍：**

网络错误 / 4xx / 5xx → banner 不显示，下次回 HomePage 再试；不打 toast。

| 维度 | 处理 |
| --- | --- |

| 速率限制 | OTP request 1/min/email；OTP verify 5 attempts/code；pair_create 5/10min/parent；pair_redeem 10/10min/device_id |

## 11. 部署拓扑

V0.5 已经是单 Vercel 项目 + python runtime + maxDuration 60s。V0.6 在此之上：

```text
happyword (Vercel project, 单)
├── api/index.py             # 不变
├── server/app/              # 新增 templates/ + static/ + parent_pages 路由
├── server/vercel.json       # 加 routes /parent/* + /static/* + /p/*
└── env vars 追加：
    EMAIL_PROVIDER            # V0.6.1，默认 "gmail_smtp"；后续可填 "resend" / "ses" 切换实现
    SMTP_HOST                 # V0.6.1，默认 smtp.gmail.com
    SMTP_PORT                 # V0.6.1，默认 587（STARTTLS）；465 用于隐式 TLS
    SMTP_USERNAME             # V0.6.1，完整 Gmail 地址（如 you@gmail.com）
    SMTP_PASSWORD             # V0.6.1，Gmail 16 位 App Password（不是登录密码）
    SMTP_FROM_EMAIL           # V0.6.1，必须 == SMTP_USERNAME（Gmail SPF/DKIM 限制）
    SMTP_FROM_NAME            # V0.6.1，From header 显示名，默认 "快乐背单词"
    SMTP_STARTTLS             # V0.6.1，默认 true（587 用 STARTTLS；465 设 false）
    PARENT_WEB_BASE_URL       # 模板中绝对链接（邮件正文 + QR payload），生产 https://happyword.vercel.app
    SESSION_COOKIE_DOMAIN     # 默认空（同 host），自定义域名时填
```

> Gmail App Password 获取流程：① 在 https://myaccount.google.com/security 启用 2-Step Verification；② 进入 https://myaccount.google.com/apppasswords 生成一个 16 位 App Password（关页面后不能再查看，必须立刻保存到 Vercel env）。一旦 Google 风控判定异常或账号关闭 2SV，App Password 即时失效；服务端 `email_provider.send` 抛 SMTP 异常时只 log + 不抛出业务错误，OTP 申请仍存 `email_verifications` 行并向 web 返回 `EMAIL_DELIVERY_DEGRADED` 提示用户稍后重试或检查垃圾箱。

`server/vercel.json` 调整：

```json
{
  "routes": [
    { "src": "/api/(.*)", "dest": "/api/index.py" },
    { "src": "/parent/(.*)", "dest": "/api/index.py" },
    { "src": "/p/(.*)", "dest": "/api/index.py" },
    { "src": "/static/(.*)", "dest": "/api/index.py" },
    { "src": "/(.*)", "dest": "/api/index.py" }
  ]
}
```

`/static/*` 由 FastAPI `StaticFiles` 服务，不走 Vercel CDN（V0.6 资源极少）。

## 12. 风险与开放问题

| 风险 | 缓解 |
| --- | --- |
| AssetStoreKit 在 emulator / 早期 NEXT 版本可能不可用 | 一律支持 preferences fallback；UI 显式提示用户 |
| QR 容量过大导致中等密度二维码读不出 | payload 设计为短 URL（≤45 字符），实测 21x21 二维码可承载；同时显示 6 位短码兜底 |
| OTP 邮件被拦截到垃圾箱 | Gmail SPF/DKIM 由 Google 自动签；@gmail.com 域 DMARC 仅 `p=none`，国内邮箱（QQ/163）误判率非零；邮件正文显式提示「未收到请检查垃圾邮件文件夹 + 添加发件地址至通讯录」；用户 5 分钟内未收到可重试 1 次 |
| email 枚举 | /request-code 总是 202；/verify-code 拒绝时不区分「邮箱不存在」与「OTP 错」 |
| 家长换设备登录发现要重新 OTP（每 30 天） | 接受；这是无密码模型的必然代价；体验好于「忘记密码 → 重置」 |
| ScanKit 进入相机界面后 user 拒绝授权 | 引导用户走 6 位短码手输路径；ScanBindingPage 里始终有「无法扫码？手动输入短码」按钮 |
| device_id fallback 路径让卸载重装后绑定失效 | 文案明示 + 家长 web 看到「设备已下线」时可手动 revoke；不视作 bug |
| 多设备同孩子并发同步 LWW 丢中间增量 | 单家庭单设备是常态；多设备明确为次要场景；V0.7+ 视实际数据决定升级 |
| **Gmail SMTP 500 封/日（普通账号）/ 2000 封/日（Workspace）滚动 24h 上限被打满** | 单家庭月级远低于阈值；监控 `email_provider.send` 异常率；接近上限时切到 Workspace 账号或经 `EMAIL_PROVIDER=resend` 切换到 Resend；触发后已存的 `email_verifications` 行不丢，web 端给用户 retry 提示 |
| **Gmail SMTP From 头必须 == SMTP_USERNAME**（伪造其他 from 会失败 SPF/DKIM 落垃圾箱） | `email_provider.py` 在初始化时 `assert SMTP_FROM_EMAIL == SMTP_USERNAME`，启动失败而不是运行时静默挂；spec §11 env vars 注释里同样写明 |
| **Gmail App Password 失效（账号关 2SV / 风控登出 / 用户手动 revoke）** | `GmailSmtpProvider.send` 捕获 `aiosmtplib.SMTPAuthenticationError` 时记 `metric_smtp_auth_fail` 并向 inbox / log 输出明显告警，OTP 申请请求仍存记录、web 返回 `EMAIL_DELIVERY_DEGRADED`；运维 runbook（V0.6.7 §17）描述「重新生成 App Password → 热更新 Vercel env → redeploy」流程 |
| **Gmail SMTP 突发并发被限速** | OTP 申请峰值预期 < 1/s；阈值升高时在 `notification_service` 加 `aiolimiter` (≤5 req/s) 串行化；不影响业务路径，仅延后投递几秒 |
| 家庭词包数量爆炸（家长创建几百个 pack） | 单 pack 50 词上限 + 客户端单 ETag 缓存 + 服务端按 family 维度可承受数千 pack；UI 在 50 个 pack 时显示 hint 「为孩子建议保持在 ≤20 个本子」，不强限 |
| 跨 pack 同 word_id 冲突 | 客户端 `mergeFamilyPacks` 用 `published_at desc, pack_id asc` 解析；hidden 强于 override；家长 web 显示 ⚠ 重复警告但不阻止 publish |
| 家庭词包总词数过大导致客户端首屏慢 | 服务端按 family 累计词数 ≥ 1000 时在 publish 响应里加 warning；客户端 mergeFamilyPacks 在合并 ≥2000 词时降级为后台延迟构建（首次冷启动用 base_repo，下一次冷启动才生效合并） |
| 家长账号被劫持后联级清空数据 | 注销 7 天 grace；删除前发邮件确认 + inbox 提示；任何阶段家长可登录 cancel |
| QR 中间人攻击（拍照 → 第三方设备扫） | token 一次性 + 3 分钟过期 + 同一 token 只能 redeem 一次；攻击者拿到的 binding 也属于自己设备，不影响家长真实设备绑定 |

**开放问题（V0.6 不解决，留 V0.7+）：**

- 多家长共享一个 family（夫妻双方都能审批）：V0.6 仅 primary_parent_id。
- 家长无法登录时孩子能看「云端愿望单」但无法兑换（除非走 PIN 即扣）：V0.6.6 接受这一短板。
- 跨家庭迁移设备：不直接支持；做法是「revoke 旧 binding → 重新扫新家长 QR」。
- 家庭词包跨 family 共享 / 模板市场：明确不做。
- 多孩子横向 leaderboard：明确不做（避免给低龄儿童制造横向比较压力）。
- 一台设备多孩子档案切换：明确不做（V0.6 模型是 1 设备 1 档案；孩子换平板就重新扫码绑）。

## 14. 已敲定的细节口径（r3 锁定）

以下 5 条原本是 r2 末尾的 "待确认 follow-up"，2026-05-01 全部敲定：

1. **OTP 邮件模板：中文 only**。邮件正文不出现英文字段名。
2. **家长 cookie 续期阈值：iat > 7 天才续**（减写入 + 减 Set-Cookie header 大小）；30 天 hard 上限触发后引导走 OTP 重登。
3. **家庭词包模型：单 family 多 pack，每包 ≤ 50 词，可无限创建**（建议每个课本单元建一个）。这条不再是「细节」，是 r3 数据模型核心，已落到 §5.7 / §6.3 / §7.6 / §8 / §9。
4. **家长账号注销 grace：7 天**（明显宽于 GDPR 最低、给「忘了取消注销」的家长留出周末时间）。
5. **家庭词包 publish 后：不向客户端推送**；客户端 HomePage `aboutToAppear` 做轻量 HEAD 检测，发现 ETag 不一致时显示非阻塞 banner「家长已更新词包，请到设置 → 同步词包」（§7.6 publish-后通知策略段）。
