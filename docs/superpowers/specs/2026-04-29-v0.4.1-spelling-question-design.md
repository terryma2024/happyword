## 1. 范围与架构概览

### 1.1 V0.4.1 一句话

> 把 V0.3 / V0.3.5 时代「点选缺失字母」（FillLetter / FillLetterMedium）这条拼写阶梯往上推一格——让 **Boss 战在词长足够时**触发**完整拼写题**：首字母揭示、剩余字母打乱后排成一排、孩子按顺序点正确字母往空位里填，**点错只抖红不扣 HP**，全部填对那一刻引擎收到 1 次 `submitAnswer` 并按 1 点伤害（或 combo burst 2 点）正常推进。

### 1.2 当前痛点（先看现状）

```text
A. 题型阶梯目前的最高档是 V0.3.5 的 FillLetterMedium（2 空两步）
   - Boss 当前题型 mix：Choice / FillLetter / FillLetterMedium，没有「写整词」的体感
   - 拼写训练强度仅到「2 空两步选择」，路线图 §4.5 显式承诺「V0.4 进入完整写单词」

B. 横屏键盘的历史顾虑（V0.3 §4.5 显式延后理由）
   - HarmonyOS 横屏调起系统键盘会遮挡战斗 UI
   - 儿童误输入成本高，键盘焦点 / 大小写 / 输入法兼容复杂
   - V0.4.1 必须**不引入系统键盘**——只用「点选式字母板」实现完整拼写

C. 现有 BattleEngine 的 submitAnswer 契约
   - Choice / FillLetter：1 题 1 次 submitAnswer
   - FillLetterMedium：1 题 2 次 submitAnswer（step 0 advancedStep:true 不扣血、step 1 完成）

```

## 2. 关键设计决策

### 2.1 Boss-only 触发 + 长度门控

```text
判定点：PlanQuestionSource 的 Boss kind 分支（QuestionGenerator 不感知 monster kind）

  word.length >= 4 AND word.length <= 9   -> SpellGenerator.generate
  word.length == 3                        -> FillLetterGenerator.generate（旧 1 空）
  word.length <= 2                        -> Choice（旧三选一）
  word.length >= 10                       -> FillLetterMedium.generate（避免 SpellingArea 长词溢出）

其他 monster kind（Normal / Spelling / Review / Elite）-> 完全不变
```

为什么是 Boss-only：
- 一局 5 怪 = 1 道 Spell 题最大暴露，孩子的「第一次写整词」体验是「打 boss 的仪式」而不是日常负担
- TodayAdventureBuilder.applyBossRotation 已确定性挑 boss + bossWordIds，0 代码改动
- 其他怪保留既有题型：Normal=Choice、Spelling=FillLetter、Review=偏错词 Choice、Elite=Choice+FillLetter mix
- 对学习强度感知：每天 1 道写整词 + N 道点选拼写 + N 道识别题，节奏不被 Spell 喧宾夺主

为什么 4-9：
- 下界 4：3 字母词一来太短（"cat"/"dog"），二来跟现有 1 空 FillLetter 重合度高，没必要再做一遍
- 上界 9：480p 横屏宽度下 9 个 slot × 48vp + 8 间距 × 8vp = 496vp，刚好放下；10+ 字母会溢出或需要折行（不在范围内）
- 13 字母词如 'extraordinary'：直接 fallback 到 FillLetterMedium（已是该词在 boss 上的兜底）

### 2.2 字母池 = 词中剩余字母（无干扰项）

```text
word = 'apple' -> letters = ['a','p','p','l','e']
                 mask    = [true, false, false, false, false]    // 仅 index 0 揭示
                 pool    = shuffle(['p','p','l','e'])             // 4 个，顺序由 RNG 决定
```

为什么不带干扰项：
- 路线图 §4.5「V0.4 再进入完整写单词」语境是「主动回忆」而非「主动辨认」——干扰项应该留 v0.4.x 后续版本
- v0.4.1 的初版定位是「让孩子第一次体验完整拼一个词」，认知负荷应该是「顺序」而不是「过滤」
- 重复字母处理简单：池子就是 multiset，孩子点任意一个 'p' 都对

### 2.3 错点反馈：抖红 + 不消耗 + 不扣 HP + 不入 wrongCount

```text
点正确字母:  letter 飞入下一空位 -> pool 该 idx 标记 consumed -> 渲染 opacity 0.2
点错字母:   letter 抖红 120ms（-8/+8/0 三段 translateX）-> pool / slots 不变
点已消费:   no-op
全部填满:   slots 整行金色高亮 200ms -> onSpellComplete() -> engine.submitAnswer(answer)
```

为什么这么宽松：
- v0.4 的目标是「孩子愿意尝试写整词」，第一次见到整词输入界面时的挫败感会直接劝退
- 5 分钟战斗倒计时已是隐性时间压力——慢吞吞拼词会自然消耗时间预算，不需要 HP 再加一道惩罚
- 对学习记录的影响有限：v0.3 wrongCount 主要驱动 review-mode gating（错词数阈值）；boss 一局 ≤1 题 Spell，无论怎么算噪音都 ≪ 其他题型

### 2.4 BattleEngine 0 改动 — Approach A

```text
SpellingArea 组件自治，BattleEngine 完全不感知字母级输入：

  TAP -> SpellingArea.onTapPoolLetter(idx)
              if 正确字母 -> 本地 state 推进
              if 错字母  -> 本地 state 抖红
              if 全部填满 -> onSpellComplete()
                              v
                       BattlePage.handleSpellSubmit()
                              v
                       engine.submitAnswer(question.answer)   // 一定 correct: true
                              v
                       outcome（damage 1 或 combo burst 2）-> 跟 Choice 路径完全同形
```

替代方案对比：
- **Approach B (engine step-by-step)**：仿 FillLetterMedium，每点一次 submitAnswer。缺点：需新增 `wasInvalidTap` outcome flag、engine 多 7-8 个单测、违反「错点不扣 HP」需要额外保护。
- **Approach C (A + telemetry)**：在 A 基础上把「首次尝试是否一次成功」写一个新字段供 v0.4.5 报告读。缺点：v0.4.5 还没设计，提前埋点是 YAGNI。

A 的核心价值：**BattleEngine.submitAnswer 契约不动 → 11 个 UI 套件 + 既有 engine 单测 0 回归风险**。

### 2.5 首字母揭示

```text
slots[0] 永远在 question 生成时就被 mask=true 揭示，pool 不含首字母
```

延续 V0.3 FillLetter / FillLetterMedium 的「永不隐藏首字母」原则：
- 早读期儿童识别第一个字母的能力波动大，揭示首字母减少认知断层
- 跟现有题型阶梯视觉一致，不引入新难度跳变
- v0.4.x 后续版本（或 v0.4.4 难度自适应）再考虑 mastered 状态下的「全空挑战」

### 2.6 视觉布局 = FillLetterMedium 同形

```
┌─────────────────────────────────────────────────────────┐
│                  [中文提示] [🔊]                          │
├─────────────────────────────────────────────────────────┤
│   [a] [_] [_] [_] [_]                                   │  <- slots 行（左对齐）
│                                                         │
│       (p)  (l)  (e)  (p)                                │  <- pool 行（居中）
└─────────────────────────────────────────────────────────┘

slot 方块：48×48 vp，圆角 8 vp，字号 28 fp 加粗
  - 揭示态：#FFF8E7 底 + #1D3557 字
  - 已填态：#EAF2F8 底 + #1D3557 字
  - 待填态：#FFFFFF 底 + #1D3557 70% 透明 _ 下划线

pool 按钮：52×52 vp 圆形，字号 24 fp 加粗
  - 默认态：#FCEAEA 底 + #E63946 字（沿用 v0.3.10 调色板）
  - 已消费：opacity 0.2，pointerEvents 关闭
  - 抖红态：#FF6B6B 底 + Curve.Sharp -8/+8/0 vp 三段 120ms 后回原色
```

视觉一致性：与 FillLetterMedium 的两行结构、配色、字号、间距规则全部对齐——孩子从 FillLetter → FillLetterMedium → Spell 的视觉迁移成本为 0，只是多了「字母候选池」的体感。

### 2.7 音频沿用 FillLetter 模式

- 中文提示 `promptZh` 顶上一行始终可见
- TTS 自动播放：`question.answer`（完整英文词）在 question 首次渲染时由 `PronunciationService.speak(answer)` 触发一次
- 🔊 重播按钮：跟 FillLetter 共用代码路径（如 FillLetter 已抽出独立 audio 子组件就直接复用，否则现写一份与之同形）
- 不引入「听音拼写」纯听模式 → v0.4.x 后续版本

### 3.1 `models/Question.ets`

新增枚举值与字段：

```typescript
export enum QuestionKind {
  Choice = 'choice',
  FillLetter = 'fill-letter',
  FillLetterMedium = 'fill-letter-medium',
  Spell = 'spell',                   // ← V0.4.1
}

export class Question {
  // 既有字段全部不动 ...

  // V0.4.1 Spell 字段
  spellLetters: string[] = [];        // 全词字母（小写），含首字母. e.g. ['a','p','p','l','e']
  spellRevealedMask: boolean[] = [];  // 与 spellLetters 等长，true=初始已揭示
  spellPool: string[] = [];           // 待选字母池（已 shuffle）. e.g. ['p','l','p','e']

  isValid(): boolean {
    if (this.wordId.length === 0) return false;
    if (this.answer.length === 0) return false;
    if (this.kind === QuestionKind.FillLetter) return this.isValidFillLetter();
    if (this.kind === QuestionKind.FillLetterMedium) return this.isValidFillLetterMedium();
    if (this.kind === QuestionKind.Spell) return this.isValidSpell();
    return this.isValidChoice();
  }

  private isValidSpell(): boolean {
    if (this.spellLetters.length < 4) return false;
    if (this.spellLetters.length > 9) return false;
    if (this.spellRevealedMask.length !== this.spellLetters.length) return false;
    if (!this.spellRevealedMask[0]) return false;       // 首字母必须揭示
    // 其余索引在 v0.4.1 必须全为 false（仅首字母揭示）
    for (let i = 1; i < this.spellRevealedMask.length; i++) {
      if (this.spellRevealedMask[i]) return false;
    }
    // pool 必须等于 letters[1..] 的多重集合
    const expected: string[] = this.spellLetters.slice(1).sort();
    const actual: string[] = this.spellPool.slice().sort();
    if (expected.length !== actual.length) return false;
    for (let i = 0; i < expected.length; i++) {
      if (expected[i] !== actual[i]) return false;
    }
    // spellLetters.join('') 必须等于 answer.toLowerCase()
    if (this.spellLetters.join('') !== this.answer.toLowerCase()) return false;
    return true;
  }
}
```

### 3.2 `services/SpellGenerator.ets`（新文件）

```typescript
import { WordEntry } from '../models/WordEntry';
import { Question, QuestionKind } from '../models/Question';
import { RandomFn, shuffle } from '../utils/shuffle';

const MIN_SPELL_LETTERS: number = 4;
const MAX_SPELL_LETTERS: number = 9;

export class SpellGenerator {
  private readonly rng: RandomFn;

  constructor(rng?: RandomFn) {
    this.rng = rng ?? Math.random;
  }

  /**
   * Generate a Spell question. Returns undefined when word length is
   * out of [MIN_SPELL_LETTERS, MAX_SPELL_LETTERS] so the caller can
   * fall back (FillLetterMedium for too-long, FillLetter / Choice for
   * too-short).
   */
  generate(word: WordEntry): Question | undefined {
    const lower: string = word.word.toLowerCase();
    const letters: string[] = [];
    for (let i = 0; i < lower.length; i++) {
      const c: string = lower.charAt(i);
      if (c >= 'a' && c <= 'z') letters.push(c);
    }
    if (letters.length < MIN_SPELL_LETTERS) return undefined;
    if (letters.length > MAX_SPELL_LETTERS) return undefined;

    const mask: boolean[] = [];
    for (let i = 0; i < letters.length; i++) {
      mask.push(i === 0);
    }

    const remaining: string[] = letters.slice(1);
    const pool: string[] = shuffle<string>(remaining, this.rng);

    const q: Question = new Question();
    q.kind = QuestionKind.Spell;
    q.promptZh = word.meaningZh;
    q.answer = word.word;
    q.options = [];
    q.wordId = word.id;
    q.spellLetters = letters;
    q.spellRevealedMask = mask;
    q.spellPool = pool;
    return q;
  }
}
```

### 3.3 `services/PlanQuestionSource.ets` 路由分支

注意：路由实际发生在 `PlanQuestionSource.nextQuestion`，**不是** `QuestionGenerator`（后者只生成 Choice 题，不感知 monster kind）。`QuestionGenerator.ets` 在 V0.4.1 完全不动。

当前 PlanQuestionSource 的 Elite 与 Boss **共享** 同一 fallback 链：

```typescript
// 当前（V0.3.5 起）
if (kind === MonsterKind.Elite || kind === MonsterKind.Boss) {
  // medium -> beginner -> choice
}
```

V0.4.1 的改动是**把 Boss 从 Elite 拆出来**，加上 SpellGenerator 优先：

```typescript
constructor(
  plan: TodaySessionPlan,
  repo: WordRepository,
  choiceGen: QuestionGenerator,
  fillGen: FillLetterGenerator,
  spellGen: SpellGenerator,        // ← V0.4.1 第 5 参
) {
  this.plan = plan;
  this.repo = repo;
  this.choiceGen = choiceGen;
  this.fillGen = fillGen;
  this.spellGen = spellGen;
  this.monsterIndexProvider = (): number => 1;
}

nextQuestion(lastWordId?: string): Question {
  const word: WordEntry = this.pickWord(lastWordId);
  const kind: MonsterKind = this.currentMonsterKind();

  if (kind === MonsterKind.Boss) {
    const spell: Question | undefined = this.spellGen.generate(word);
    if (spell !== undefined) return spell;
    const medium: Question | undefined = this.fillGen.generateMedium(word);
    if (medium !== undefined) return medium;
    const beginner: Question | undefined = this.fillGen.generate(word);
    if (beginner !== undefined) return beginner;
    return this.choiceGen.nextQuestionForWord(word, lastWordId);
  }
  if (kind === MonsterKind.Elite) {
    const medium: Question | undefined = this.fillGen.generateMedium(word);
    if (medium !== undefined) return medium;
    const beginner: Question | undefined = this.fillGen.generate(word);
    if (beginner !== undefined) return beginner;
    return this.choiceGen.nextQuestionForWord(word, lastWordId);
  }
  if (kind === MonsterKind.Spelling) {
    const fill: Question | undefined = this.fillGen.generate(word);
    if (fill !== undefined) return fill;
    return this.choiceGen.nextQuestionForWord(word, lastWordId);
  }
  return this.choiceGen.nextQuestionForWord(word, lastWordId);
}
```

要点：
- Boss 分支：`spell -> medium -> beginner -> choice`，长度门控隐式由 generator 各自的 `undefined` fallback 处理（spell 限 4-9、medium 限 ≥4、beginner 限 ≥3）
- Elite 分支：**完全保持 V0.3.5 行为**，与 Boss 不再共享 if-OR
- Spelling / Normal / Review：完全不变
- 构造函数签名变化是破坏性 API 变更，所有调用 PlanQuestionSource 构造的位置必须同步加 `spellGen` 参数（BattlePage 与可能存在的测试夹具）

调用方迁移：
- `BattlePage.aboutToAppear`（或同等位置）：构造 PlanQuestionSource 时多传一个 `new SpellGenerator()`
- 既有单元测试构造 PlanQuestionSource 时同步加参

### 3.4 `components/SpellingArea.ets`（新文件）

```typescript
@Component
export struct SpellingArea {
  @Prop question: Question;
  onSpellComplete: () => void = () => {};

  @State private slots: string[] = [];
  @State private consumedPoolIdx: boolean[] = [];
  @State private shakingPoolIdx: number = -1;
  @State private completionFlash: boolean = false;

  aboutToAppear(): void {
    this.resetForQuestion();
  }

  // ArkUI 不能直接 @Watch prop 对象的内部变化，但 question 整体替换会触发组件重渲染。
  // 实践：父组件在切题时给 @Prop 赋新引用，触发 aboutToAppear / 重新 init。
  // 若 ArkUI 实测仍需 @Watch('question')，按需补一个空-参 watcher 调 resetForQuestion。

  private resetForQuestion(): void {
    const q: Question = this.question;
    const slots: string[] = [];
    for (let i = 0; i < q.spellLetters.length; i++) {
      slots.push(q.spellRevealedMask[i] ? q.spellLetters[i] : '_');
    }
    this.slots = slots;
    const consumed: boolean[] = [];
    for (let i = 0; i < q.spellPool.length; i++) consumed.push(false);
    this.consumedPoolIdx = consumed;
    this.shakingPoolIdx = -1;
    this.completionFlash = false;
  }

  private onTapPoolLetter(poolIdx: number): void {
    if (this.consumedPoolIdx[poolIdx]) return;
    const letter: string = this.question.spellPool[poolIdx];
    const nextSlot: number = this.firstEmptySlotIndex();
    if (nextSlot < 0) return;
    const expected: string = this.question.spellLetters[nextSlot];
    if (letter === expected) {
      this.slots[nextSlot] = letter;
      this.consumedPoolIdx[poolIdx] = true;
      // ArkUI 数组深响应：复制再赋值确保触发 rerender
      this.slots = this.slots.slice();
      this.consumedPoolIdx = this.consumedPoolIdx.slice();
      if (this.firstEmptySlotIndex() < 0) {
        this.completionFlash = true;
        setTimeout((): void => {
          this.completionFlash = false;
          this.onSpellComplete();
        }, 200);
      }
    } else {
      this.shakingPoolIdx = poolIdx;
      // 抖动动画通过 animateTo + state flip 实现，120ms 后清空
      setTimeout((): void => { this.shakingPoolIdx = -1; }, 140);
    }
  }

  private firstEmptySlotIndex(): number {
    for (let i = 0; i < this.slots.length; i++) {
      if (this.slots[i] === '_') return i;
    }
    return -1;
  }

  build() {
    Column() {
      // slots 行
      Row() {
        ForEach(this.slots, (slot: string, i: number) => {
          // 渲染 48×48 vp 方块；揭示 / 已填 / 待填三态分支
        });
      }
      // pool 行
      Row() {
        ForEach(this.question.spellPool, (letter: string, i: number) => {
          // 渲染 52×52 vp 圆形按钮；consumed / shake / 默认三态
          Button(letter)
            .id(`SpellPoolBtn-${i}`)
            .opacity(this.consumedPoolIdx[i] ? 0.2 : 1)
            .onClick(() => this.onTapPoolLetter(i))
        });
      }
    }
  }
}
```

实现要点：
- `slots` 与 `consumedPoolIdx` 用 `slice()` 后再赋值，确保 ArkUI 触发响应（参考既有 SpellingArea 类似组件的状态深拷贝模式）
- `shakingPoolIdx` 用 `animateTo` 三段 translateX；不要用 setInterval
- 按钮 id 命名 `SpellPoolBtn-{idx}`、slot id 命名 `SpellSlot-{idx}` —— UI 测试通过这俩 id 抓字母
- `onSpellComplete` 由父组件传入，父组件负责调 `engine.submitAnswer`

`handleSpellSubmit()`：

```typescript
private handleSpellSubmit(): void {
  const q: Question | undefined = this.engine.getState().currentQuestion;
  if (q === undefined || q.kind !== QuestionKind.Spell) return;
  const outcome: AnswerOutcome = this.engine.submitAnswer(q.answer);
  // 既有 outcome 处理路径完全复用：
  this.applyOutcomeAnimations(outcome);   // combo burst, monster hit, projectile
  this.recordIfWordMutationNeeded(q.wordId, outcome.correct);
  this.advanceOrEndBattle(outcome);
}
```

注意：`engine.submitAnswer` 在 Spell 路径上一定收到正确答案 → outcome.correct === true（除非 boss 战时被 race 条件干扰，理论不发生）。所以 spell 完成的视觉反馈和 Choice 完全同形，没有特殊路径。

## 4. 数据迁移 / 持久化兼容性

## 6. 风险 & 缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| 词包中 boss 词长度分布偏短 → Spell 暴露率低 | v0.4.1 价值感不足 | 现有 7 个 boss 词 + Forest/Castle/Cottage 区域词包预审；若某 region 实际 < 4 字母 boss 词占比过高，调整 boss word filter 让 4-9 字母词优先 |
| ArkUI 数组 @State 深响应踩坑 | slots / consumedPoolIdx 不触发 rerender | 强制使用 `.slice()` 后整体赋值；SpellingArea.test.ets 直接覆盖此路径 |
| 抖动动画与 Curve.Sharp 在 HarmonyOS NEXT 真机表现不一致 | 视觉反馈不达预期 | 备选方案：用 setTimeout 三段 translateX 手工帧动画（即设三次 -8/+8/0），代价是稍粗糙但稳定 |
| 长词 fallback 让某些 boss 永远不出 Spell | 期望感受不到 v0.4.1 改造 | hard cap 9 字母在词包语境下足够宽松（统计现有词典 4-9 字母占比 >70%）；若实测有 region 卡 cap，再单点放宽至 10 字母 |
| BattleEngine.submitAnswer 在 Spell 完成时收到非 currentQuestion.answer 的字符串（race） | 抛 InvalidOption 错误 | SpellingArea 在 onSpellComplete 调用时锁定当前 question 引用；若 race 发生，BattlePage 的 handleSpellSubmit 先 check kind === Spell 再调 submitAnswer |
| 测试夹具难以注入「force boss = apple」 | UI 测试覆盖率不足 | 沿用 v0.3.8 boss override 路径：通过 AppStorage 注入测试用 monsterSlots 和 bossWordIds（如已存在该 hook，直接复用） |

## 8. V0.4.1 不做（YAGNI 收紧）

- ❌ 不做"听音拼写"纯听模式（隐藏中文提示）→ v0.4.x 后续
- ❌ 不做 Spell 题首试命中率 telemetry → v0.4.5 报告时再补（方案 C 增量）
- ❌ 不做"提示字母"按钮（点一下解锁下一空位）→ 难度自适应留 v0.4.4
- ❌ 不做 SpellPad 干扰字母（带 distractors 的池子）→ v0.4.x 升级时再说
- ❌ 不做长词的 SpellPad 折行 / 缩放手势 → 9 字母 hard cap 简单粗暴够用
- ❌ 不做 Spell 专属 SFX / 完成欢呼动画 → 复用现有 combo burst / monster-hit 动效
- ❌ 不改 LearningRecorder / MemoryScheduler → 错点不入 wrongCount，准确度问题留 v0.4.3 再说
- ❌ 不在 MonsterCodexPage 增加"该 boss 出 Spell"标识 → codex 跟战斗题型解耦
- ❌ 不动 v0.3.5+ 的 FillLetter / FillLetterMedium 任何代码 → 它们继续服务 Spelling/Elite 槽
- ❌ 不引入新词包 / 新 region / 新 boss → 用现有 7 boss + 3 region 数据
- ❌ 不引入系统键盘 / 自定义字母键盘（26 键）→ 只做 scrambled pool
- ❌ 不做 Spell 完成的额外 coin 奖励 → 跟 Choice / FillLetter 同酬
- ❌ 不持久化 Spell 进度（中途退出战斗 = 当前 question 丢弃，跟其他题型一致）
