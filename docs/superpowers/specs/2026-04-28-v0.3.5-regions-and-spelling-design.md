## 2. 数据模型

### 2.1 题型扩展 — `Question.kind = FillLetterMedium`

```text
enum QuestionKind {
  Choice = 'choice',
  FillLetter = 'fill-letter',
  FillLetterMedium = 'fill-letter-medium',  // V0.3.5 新增
}

class Question {
  // ... V0.3 现有字段不动 ...

  // V0.3.5 中级题专用 (其他 kind 留空)
  letterTemplateBase: string = ''            // 例 'a _ p _ e' (与 V0.3 letterTemplate 同形态)
  missingIndices: number[] = []              // 长度 2; 步顺序; 升序非必须
  letterOptionsSteps: string[][] = []        // 长度 2; 每元素是该步的 3 个候选
  letterAnswers: string[] = []               // 长度 2; 各步正确字母 (lower-case)
  currentStep: number = 0                    // 引擎运行时状态; 0 = 第 1 步
}
```

`isValid()` 按 `kind` 分派；新增 `isValidFillLetterMedium()`：
- `letterTemplateBase` 非空。
- `missingIndices.length === 2`，两元素互不相等，且都 `>= 1`（避开词首字母）且 `< answer.length`。
- `letterOptionsSteps.length === 2`；每个子数组长度 3、单字符、互不重复。
- `letterAnswers.length === 2`；每元素是单字符 lower-case；`letterOptionsSteps[i].indexOf(letterAnswers[i]) >= 0`；`answer.toLowerCase()[missingIndices[i]] === letterAnswers[i]`。
- `currentStep` 在 `[0, missingIndices.length - 1]`。

## 3. 服务层

### 3.1 `FillLetterGenerator.generateMedium`（新方法）

签名：

```text
class FillLetterGenerator {
  generate(word: WordEntry, lastWordId?: string): Question | undefined  // V0.3 不动
  generateMedium(word: WordEntry, lastWordId?: string): Question | undefined  // V0.3.5 新
}
```

规则：

1. 取 `letters = lower-cased alphabetic chars`；若 `letters.length < 4`，返回 `undefined`（让上层退回 beginner FillLetter，再退回 Choice）。
2. 选 `idx0 ∈ [1, letters.length - 1]`，再从 `[1, letters.length - 1] \ {idx0}` 选 `idx1`。两个索引的呈现顺序 = 数组下标顺序，**不强制按字面顺序由左到右**（避免可预测性；UI 通过当前 step 高亮指示哪个空当前要填）。
3. 每步独立选 2 个干扰字母：从 `'abcdefghijklmnopqrstuvwxyz'` 中剔除当前步答案；优先剔除该词内出现的所有字母与另一步的答案；若候选 < 2，回退到只剔除当前步答案的池。
4. `letterOptionsSteps[i] = shuffle([letterAnswers[i], distractors[0], distractors[1]])`。
5. `letterTemplateBase` 同 beginner 形态：`'a _ p _ e'`（**两个空都用 `_` 渲染**；BattlePage 用 `currentStep`/`missingIndices` 决定哪个空高亮）。
6. 写入 `Question { kind: FillLetterMedium, answer: word.word, wordId: word.id, letterTemplateBase, missingIndices, letterOptionsSteps, letterAnswers, currentStep: 0 }`。

不变性：同一 `rng` 种子产同一题。短词（长度 < 4，例如 `bed/cup/zoo/TV`）失败用 `undefined` 表示。

### 3.2 `PlanQuestionSource` 路由调整

`nextQuestion` 内部按 `currentMonsterKind()` 路由：

```text
kind === Boss   -> generateMedium 优先; 失败回 generate; 再失败回 choice
kind === Elite  -> generateMedium 优先; 失败回 generate; 再失败回 choice
kind === Spelling -> generate (beginner); 失败回 choice
kind === Review | Normal -> choiceGen.nextQuestionForWord
```

> 与 V0.3 区别：V0.3 仅 `Spelling | Boss` 走 fillGen，`Elite` 走 choice。V0.3.5 把 `Elite` 也升级到 fill-letter 路径（中级），让"精英怪比拼写怪更难"在题型上立得住。Boss 视觉同 Dragon、Elite 视觉同 Dragon，但题型偏好上 Boss 与 Elite 都是 medium。

### 3.3 `BattleEngine.submitAnswer` 原子双步分支

`AnswerOutcome` 追加：

```text
class AnswerOutcome {
  // ... V0.3 字段不动 ...
  advancedStep: boolean = false   // V0.3.5 新增; 仅当 medium 题非末步答对时为 true
}
```

`submitAnswer(option)` 在 `q.kind === FillLetterMedium` 时按以下规则：

1. `validOptions = q.letterOptionsSteps[q.currentStep]`；不在其内 → 抛错（与 V0.3 一致）。
2. `correct = option === q.letterAnswers[q.currentStep]`。
3. **错答**（任一步错即整题错）：走原"错答"路径——`comboCount = 0`、`playerHp -= 1`、`totalAnswers += 1`、判定是否 Lost、轮换下一题。`outcome.advancedStep = false`。
4. **对答且非末步**：
    - `q.currentStep += 1`。
    - **不修改** `comboCount` / `monsterHp` / `totalAnswers` / `correctAnswers` / `learnedWordIds` / `defeatedMonsters` / `state.currentQuestion`。
    - **不调用** `generator.nextQuestion`。
    - 返回 `outcome { correct: true, damage: 0, advancedStep: true, comboTriggered: false, monsterDefeated: false, newMonsterSpawned: false, battleEnded: false }`。
5. **对答且末步**：走原"对答"路径——`comboCount += 1`、判 burst、扣怪 hp、判杀怪/胜利、轮换下一题。`outcome.advancedStep = false`。

> 设计要点：medium 题在非末步答对时**不算一次"完整答题"**，因此 `totalAnswers` 与 `correctAnswers` 都不增。这样 `correctRate` 与星级计算保持"题级"语义，与玩家直觉一致。Choice 与 beginner FillLetter 的语义、单测全部不变。

### 3.4 BattleEngine 单测覆盖

引擎层新增 5 个单测：
1. medium 第 1 步对 → `outcome.advancedStep === true`、`state.currentQuestion === q`、所有计数与 hp 不变。
2. medium 末步对 → 走完整正确路径（伤害 1 或 burst 触发时 2）、轮换下一题、`outcome.advancedStep === false`。
3. medium 第 1 步错 → 走完整错答路径、玩家扣血、combo 归零、`outcome.advancedStep === false`、`state.currentQuestion` 已轮换。
4. medium 末步错 → 同 3（保证"前一步对"事实不影响最终判负）。
5. medium 中途时 burst 阈值：第 2 步对触发 burst，伤害 = 2 与现有 burst 单测一致。

`letterSlots()` 扩展（保持 codelinter 友好；先把 `this.letterMissingIndex` / `this.isMedium` / `q.missingIndices` 拷到本地常量再迭代）：

- 非 medium：与 V0.3 行为一致，仅高亮 `letterMissingIndex`。

1. 仍调 `engine.submitAnswer(text)` 取 `outcome`。
2. 若 `outcome.advancedStep === true`：
    - `this.currentStep = q.currentStep` 同步（引擎已自增）。
    - 反馈窗口期间 prompt 不变；UI 在该步显示一个微提示文本"答对啦！下一空"，使用现有 `feedbackText` 状态。
    - 反馈结束后 `syncFromState()` 重读 `q.letterOptionsSteps[currentStep]` 与 `q.missingIndices[currentStep]` 重渲选项；不播 `HIT_CRIT`、不增 combo 视效；播一个轻反馈（复用 `SoundKeys.HIT_NORMAL`）。
3. 否则（末步对、或任一步错）：维持现有正确/错误反馈链路（包含可能的 burst → `HIT_CRIT`）。

> 注意：medium 反馈期长度沿用 V0.3 既有 `FEEDBACK_MS = 650`（[`BattlePage.ets:86`](../../../entry/src/main/ets/pages/BattlePage.ets)），不引入新的时序常量。

### 4.3 BattlePage 怪物视觉解析

新增私有 helper：

```text
private resolveMonsterEntry(state: BattleState): MonsterEntry {
  if (this.isTodayAdventure && this.todayPlan) {
    const slots = this.todayPlan.monsterSlots
    const idx = state.monsterIndex <= 0 ? 0 : (state.monsterIndex - 1) % slots.length
    return getMonsterByIndex(slots[idx].catalogIndex)
  }
  return getMonsterByIndex(state.monsterIndex)
}
```

`syncFromState()` 中把 `this.currentMonster = getMonsterByIndex(state.monsterIndex)` 改为 `this.currentMonster = this.resolveMonsterEntry(state)`。

## 6. 失败与降级路径

| 场景 | 处理 |
|------|------|
| `TodayPreferences.loadRegionId` 抛错或返回非法 id | 退回 `DEFAULT_REGION_ID`，`console.error` 记录 |
| AppStorage 中 `TODAY_REGION_ID_KEY` 为非法 id | `getRegionById` 已经兜底返回 `[0]` |
| 短词无法生成 medium | `FillLetterGenerator.generateMedium` 返回 `undefined`，PlanQuestionSource 退到 beginner FillLetter；再不行退 Choice |
| 主题分类下没有词 | `gatherCandidates` 已有 fallback 到全词库（V0.3 行为不变） |
| `hit_crit.ogg` 资源缺失 | `AudioService` 既有"缺失静默"分支不变 |
| medium 题在 BattlePage 反馈窗口期间收到时间 tick → Lost | 沿用 V0.3 行为：`tick` 走 Lost 路径，`outcome.battleEnded` → 路由结算页；未结算的 currentStep 不持久化 |
| `rawfile/character/*.svg` 缺失或损坏 | ArkUI `Image` 组件渲染空白方块 + 控制台报错（`could not load resource ...`）；不会崩溃，但视觉损失明显。验收阶段必须人工核对 4 张图能正常显示 |
| 设备 ArkUI 版本不支持 SVG `linearGradient` | 极少数老 SDK 才会发生；HarmonyOS NEXT 主线版本均已支持。若发生，回滚整个 V0.3.5 SVG 改动到形状原语（git revert SVG 任务的 commit 即可，由于 §1.4 已声明"无兜底分支"，需要手动恢复 mageAvatar/slimeAvatar） |

## 7. 与路线图的差异说明

- 路线图 §5.1 描述中级题"需要 UI 支持两步状态、**错选撤销**和'差一步'提示"。本 spec 明确**采用无撤销方案**（用户在 brainstorming 阶段确认 `stepwise no undo`），即任一步错即整题错。理由：
    1. 撤销引入"已对的步是否清零"的二义性；
    2. 与 BattleEngine 现有"一次提交=一次结算"的语义对齐；
    3. 与"原子双步"语义保持一致（任意一错即整题错，不允许部分得分）。
- 路线图 §4.4 表中 Boss 列写"混合题型"，V0.3.5 现把 Elite/Boss 都偏 medium。这是"在 V0.3 实现 spelling=Boss 偏好"基础上的进一步收紧；如果发现 medium 触发率过高再在 V0.4 调比例（YAGNI）。
- 路线图 §11 把"角色立绘升级"放在 V0.8 Cocos2D 重写阶段。本次提前用 ArkUI `Image($rawfile)` 渲染 4 张 SVG 立绘，是**ArkUI 内的视觉升级**，不是引擎切换：动画系统仍是 ArkUI `animateTo`，资产仍是 rawfile，未引入 Cocos Creator 任何能力；所以与 V0.8 计划不冲突——V0.8 仍计划完整重写战斗表现层。

## 8. V0.3.5 明确不做

- 不引入键盘式拼写、不上撤销/重试、不引入"差一步"半对半奖励。
- 不调整 `monstersTotal=5` / `monsterMaxHp=3` / `playerMaxHp=5` / `startingSeconds=300` 等战斗常量。
- 不调整魔法币上限 / 愿望单 catalog / 长按 3 秒确认。
- 不修改 `LearningSnapshot` schema（保持 v2）；不增 `MemoryState` 状态。
- 不重写 `BattlePage` 大型结构；中级题渲染只在 `letterSlots()`、`syncFromState()`、`onOptionTap` 三个点局部加分支。
- 不接服务端、不接账号、不引入新词库。
- 不为 SVG 角色加序列帧/Spine 动画——立绘是静态 PNG 替身，所有动作仍由 ArkUI `Stack` 上的 scale/translate/rotate 驱动。
- 不引入 SVG 主题切换或运行时着色；同一 kind 的所有怪物共享同一张 SVG（与 §2.2 共享视觉一致）。
- 不保留形状原语兜底；如 SVG 加载失败，视觉损失由 §6 失败矩阵承接。
