# V0.7.0 Monorepo Native Clients Restructure - Migration Design

- **Date:** 2026-05-06
- **Status:** Design-for-implementation
- **Roadmap:** [WordMagicGame_roadmap.md §14 V0.7.0](../../WordMagicGame_roadmap.md)
- **Scope:** Repository and project-layout migration only. No gameplay, API, server, or client feature behavior changes.
- **Target shape:** Root-level `harmonyos/`, `ios/`, `android/`, `server/`, `shared/`.
- **Primary constraint:** DevEco Studio must migrate smoothly from the current repository root project to `harmonyos/` as the new HarmonyOS project root.

## 1. Background

The repository started as a HarmonyOS NEXT project and later gained a real `server/` backend, deployment tooling, product specs, assets, and automation. The root directory is currently both:

- the Git / Codex workspace root; and
- the DevEco Studio HarmonyOS project root.

That dual identity is becoming expensive now that WordMagicGame is moving toward three native child clients: HarmonyOS, iOS, and Android. If the root remains a DevEco project, every root-level addition looks like noise to DevEco and every Harmony build command assumes repo root as project root.

V0.7.0 solves this by converting the repo into a monorepo while preserving a complete DevEco-managed HarmonyOS project under `harmonyos/`.

## 2. Goals

- Make the repository root a stable monorepo workspace.
- Move the HarmonyOS project into `harmonyos/` as a full DevEco Studio project root.
- Keep `server/` at root as the FastAPI backend module.
- Reserve root-level `ios/` and `android/` for native implementations: Swift / SwiftUI and Kotlin / Jetpack Compose.
- Introduce `shared/` only for contracts, schemas, fixtures, and protocol docs; no shared runtime client framework.
- Keep the existing HarmonyOS app behavior unchanged after the move.
- Preserve the existing asset retention policy.
- Update build, lint, test, CI, and agent instructions to use the new working directories.

## 3. Non-Goals

- Do not start the iOS or Android product implementation in this migration.
- Do not introduce Flutter, React Native, Unity, or another cross-platform runtime.
- Do not change server API behavior.
- Do not refactor ArkTS services or pages unless a path move forces import or resource reference fixes.
- Do not delete retired assets; move design-source or binary resources under `assets/` if anything becomes unused.
- Do not replace DevEco / Hvigor project structure with a custom build layout.

| Path | Owner | Purpose |
| --- | --- | --- |
| `harmonyos/` | HarmonyOS client | Complete DevEco Studio project root for ArkTS / ArkUI app. |
| `ios/` | iOS client | Native iOS app. Swift / SwiftUI when implementation starts. |
| `android/` | Android client | Native Android app. Kotlin / Jetpack Compose when implementation starts. |

| `assets/` | Durable source assets | Design-source SVGs, audio sources, screenshots, retired visual variants. |
| `docs/` | Product and engineering docs | Roadmap, specs, plans, runbooks. |
| `tools/` | Repo tooling | Asset generation, Vercel helpers, migration helpers. |
| `scripts/` | Root orchestration scripts | Only scripts that intentionally coordinate multiple modules. |

## 6. HarmonyOS DevEco Migration Contract

The HarmonyOS move must treat DevEco project identity as first-class. These files move together into `harmonyos/`:

```text
AppScope/
entry/
hvigor/
hvigorfile.ts
build-profile.json5
oh-package.json5
oh-package-lock.json5
code-linter.json5
```

The existing `build-profile.json5` contains:

```json5
"modules": [
  {
    "name": "entry",
    "srcPath": "./entry"
  }
]
```

After moving `build-profile.json5` and `entry/` under the same `harmonyos/` directory, `srcPath: "./entry"` remains correct. This is the main reason to move the Harmony project as a whole rather than splitting `entry/` away from its Hvigor project files.

Local-only DevEco files need a separate handling rule:

| Current path | Target handling | Commit? |
| --- | --- | --- |
| `local.properties` | Move or recreate as `harmonyos/local.properties`. Required for local SDK path. | No |
| `.idea/` | Prefer regenerate by opening `harmonyos/` in DevEco. Copy only if preserving IDE window state is worth the cache risk. | No |
| `.hvigor/` | Do not move; regenerate from `harmonyos/`. | No |
| `oh_modules/` | Do not move; regenerate with `cd harmonyos && ohpm install`. | No |
| `build-tmp/` | Do not move; delete or ignore as transient build output if present. | No |

Recommended DevEco handoff:

1. Close the currently opened root project in DevEco Studio.
2. Run the git-level migration.
3. Ensure `harmonyos/local.properties` exists.
4. Open `/Users/bytedance/Projects/happyword/harmonyos` in DevEco Studio.
5. Let DevEco re-index and regenerate `.idea/`, `.hvigor/`, and previews.
6. Run `ohpm install`, then `hvigorw assembleHap` from `harmonyos/`.

This avoids carrying stale root-level `.idea/.deveco` module caches into the new project root.

## 7. Root-Level Files After Migration

Root-level `AGENTS.md` remains the monorepo guide. It should no longer describe root as the HarmonyOS project root. Instead, it should state:

- HarmonyOS work lives under `harmonyos/`.
- HarmonyOS commands must run from `harmonyos/`.
- Server commands must run from `server/`.
- iOS and Android are native-client roots and should not depend on cross-platform runtime code.

`CLAUDE.md` should mirror `AGENTS.md` if it remains a project instruction surface.

Root `README.md` becomes a monorepo overview with per-module command snippets.

`.cursor/dev-commands.md` can either stay at root as the source of truth or move to `harmonyos/.cursor/dev-commands.md`. For the first migration, keep it at root to minimize IDE and agent churn, but update every Harmony command to say `Working directory: harmonyos/`.

## 8. Command Matrix

### HarmonyOS

```sh
cd harmonyos
ohpm install
hvigorw assembleHap
codelinter -c ./code-linter.json5 . --fix
hvigorw --mode module -p module=entry assembleHap
hvigorw -p module=entry@default test
```

UI tests continue to require an emulator or device:

```sh
cd harmonyos
hdc list targets
../scripts/run_ui_tests.sh
```

The UI script may need path updates because it currently coordinates the Harmony app, `server/mock_ui_server.py`, `hdc rport`, and `aa test`.

### Server

```sh
cd server
uv run pytest
uv run ruff check .
uv run ruff format --check .
uv run mypy app
```

Server discipline remains unchanged: every commit touching `server/` must run `uv run pytest` with 0 errors and 0 warnings.

### iOS

V0.7.0 creates only the root placeholder. Future implementation should use Swift / SwiftUI and XCTest.

```sh
cd ios
# future: xcodebuild test ...
```

### Android

V0.7.0 creates only the root placeholder. Future implementation should use Kotlin / Jetpack Compose and Gradle.

```sh
cd android
# future: ./gradlew test ...
```

## 9. Shared Contracts

`shared/` is intentionally not a shared client SDK. It contains durable protocol artifacts:

```text
shared/
  contracts/
    openapi.json
    errors.md
    word-pack.schema.json
    sync.schema.json
    device-binding.schema.json
  fixtures/
    word-packs/
    learning-stats/
    sync/
    api/
  README.md
```

Rules:

- Server owns the canonical API behavior.
- `shared/contracts/` records snapshots and schemas that native clients test against.
- HarmonyOS, iOS, and Android may generate local models from schemas later, but no shared runtime logic is required.
- Golden fixtures should cover word-pack parsing, family-pack merge, learning-stat merge, and device-binding payloads.

## 10. Migration Phases

### Phase 1 - Documentation and Safety Prep

- Add this V0.7.0 design document.
- Update `WordMagicGame_roadmap.md` so V0.7.0 is the monorepo/native-client restructure.
- Update root `README.md`, `AGENTS.md`, and `CLAUDE.md` to describe the new target working directories.
- Update `.cursor/dev-commands.md` command working directories.
- No files are moved in this phase.

Exit criteria:

- The migration target is documented.
- Every known command has a new intended working directory.
- DevEco handoff steps are explicit.

### Phase 2 - Create Root Module Skeletons

- Create `harmonyos/`, `ios/`, `android/`, and `shared/`.
- Add placeholder README files for `ios/`, `android/`, and `shared/`.
- Do not create native iOS or Android project files yet unless that work is explicitly scheduled.

Exit criteria:

- Root-level future module names exist.
- No Harmony build behavior has changed yet.

### Phase 3 - Move HarmonyOS Project

Move tracked Harmony project files into `harmonyos/`:

```text
AppScope/              -> harmonyos/AppScope/
entry/                 -> harmonyos/entry/
hvigor/                -> harmonyos/hvigor/
hvigorfile.ts          -> harmonyos/hvigorfile.ts
build-profile.json5    -> harmonyos/build-profile.json5
oh-package.json5       -> harmonyos/oh-package.json5
oh-package-lock.json5  -> harmonyos/oh-package-lock.json5
code-linter.json5      -> harmonyos/code-linter.json5
```

Local-only handoff:

```text
local.properties       -> harmonyos/local.properties   # local only
```

Do not move:

```text
.idea/
.hvigor/
oh_modules/
build-tmp/
```

Exit criteria:

- `cd harmonyos && ohpm install` succeeds.
- `cd harmonyos && hvigorw assembleHap` succeeds.
- `cd harmonyos && codelinter -c ./code-linter.json5 . --fix` succeeds after build.

### Phase 4 - Repair Paths and Scripts

Update path-sensitive references:

- `.cursor/dev-commands.md`
- root `AGENTS.md` and `CLAUDE.md`
- root `README.md`
- `scripts/run_ui_tests.sh`
- `tools/recraft/*` scripts that read or write Harmony rawfiles
- `tools/generate_scan_binding_qr_fixture.py` if any resource output path assumes root `entry/`
- docs that link directly to `entry/src/...`, `AppScope/...`, or root Hvigor files

Path examples:

```text
entry/src/main/resources/rawfile/...       -> harmonyos/entry/src/main/resources/rawfile/...
AppScope/resources/...                     -> harmonyos/AppScope/resources/...
code-linter.json5                          -> harmonyos/code-linter.json5
build-profile.json5                        -> harmonyos/build-profile.json5
```

Exit criteria:

- `rg "entry/src|AppScope|code-linter.json5|build-profile.json5|oh-package" docs scripts tools AGENTS.md CLAUDE.md README.md .cursor` has no stale command-critical references.
- Root docs clearly distinguish repo root from HarmonyOS project root.

### Phase 5 - Restore Test and Automation Pipeline

- Run Harmony build.
- Run Harmony CodeLinter.
- Run Harmony no-device unit tests.
- Run server tests if any server files changed.
- Run UI test script only after the path updates are complete and a device/emulator is available.

Exit criteria:

- Harmony HAP builds from `harmonyos/`.
- Local unit tests still find `@ohos/hypium` through `harmonyos/oh_modules`.
- UI script can still boot the host mock server and run `aa test`.
- Server test expectations are unchanged.

### Phase 6 - Commit Boundary

Use one migration commit for the move and path repairs. Do not mix iOS/Android feature work into that commit.

Recommended commit message:

```text
refactor(repo): move HarmonyOS project under harmonyos
```

If the move becomes too large, split into:

1. `docs(roadmap): define V0.7.0 monorepo restructure`
2. `chore(repo): add native client module placeholders`
3. `refactor(harmonyos): move DevEco project under harmonyos`
4. `chore(tools): update monorepo paths`

## 12. Risk Register

| Risk | Impact | Mitigation |
| --- | --- | --- |
| DevEco keeps stale root-project cache | It may open the old root as if it were still the Harmony project. | Close DevEco, open `harmonyos/` explicitly, regenerate `.idea/`, keep old `.idea` uncommitted. |
| `local.properties` missing after move | Hvigor cannot locate SDK. | Move or recreate `harmonyos/local.properties` locally; keep it ignored. |
| `oh_modules/` missing | Unit tests may hang or fail to register Hypium. | Run `cd harmonyos && ohpm install` before build/test. |
| Path-sensitive UI scripts fail | UI automation cannot find HAP or mock fixtures. | Update `scripts/run_ui_tests.sh` and fixture paths in the same migration commit. |
| Generated asset tools write to old rawfile path | Icons/audio/characters generated into dead paths. | Update `tools/recraft/*` to target `harmonyos/entry/src/main/resources/rawfile/...`. |
| Docs retain stale root-Harmony commands | Agents run commands from wrong directory. | Update `.cursor/dev-commands.md`, `AGENTS.md`, `CLAUDE.md`, and root README first. |
| iOS/Android placeholders imply feature scope | Migration balloons into new client implementation. | Placeholder READMEs only in V0.7.0; native implementation gets separate specs. |

## 13. Rollback Strategy

If Harmony build or DevEco project opening fails after the move:

1. Do not change server, iOS, or Android directories while debugging.
2. Verify `harmonyos/build-profile.json5`, `harmonyos/oh-package.json5`, and `harmonyos/hvigorfile.ts` are present.
3. Recreate `harmonyos/local.properties`.
4. Remove local `harmonyos/.hvigor/` and rerun `ohpm install`.
5. If DevEco still treats root as project root, close the IDE and open `harmonyos/` from the welcome screen.
6. If the move must be reverted, revert the migration commit as a unit; do not hand-move partial files back.

## 14. Future Native Client Work

After V0.7.0 is complete, iOS and Android should each get their own design docs before implementation.

Recommended order:

1. `V0.7.1 shared contracts and fixtures` - extract OpenAPI/schema/fixtures from the current server and Harmony expectations.
2. `V0.7.2 Android native walking skeleton` - Kotlin / Jetpack Compose, word-pack fetch + offline cache + minimal battle loop.
3. `V0.7.3 iOS native walking skeleton` - Swift / SwiftUI, same contract surface as Android.
4. Later versions can fill parity: account binding, sync, wishlist approval, audio, and richer battle presentation.

Native clients should share protocols and fixtures, not UI code or runtime business logic.
