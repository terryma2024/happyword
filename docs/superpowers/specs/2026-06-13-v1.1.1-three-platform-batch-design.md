**Scope:** HarmonyOS + iOS + Android (and the shared Cocos battle scene)
**Process:** Lightweight coordinated batch (not the full three-platform feature SOP). Implemented in parallel across platforms, version bumped to v1.1.1, pushed to `main`.

## Summary

Three independent optimizations shipped together in v1.1.1:

1. **Daily coin cap 20 → 100** on all three native clients.
2. **Sentence-cloze TTS reads the full English sentence** (with the answer filled into the blank) instead of only the answer word.

## Item 2 — Sentence-cloze TTS reads the full English sentence

### Current behavior
The 5th question type, `sentence-cloze`, shows a sentence with a `____` blank. Tapping the speaker currently speaks **only the answer word** (`question.answer`) on all three platforms. Auto-speak is intentionally disabled for this question type.

### Desired behavior
Tapping the speaker speaks the **complete English sentence with the answer filled in** — e.g. `"The cat sat on the mat"`. The `____` placeholder must **not** be vocalized; it is replaced by the answer word before the string reaches TTS. Only the `sentence-cloze` type changes; all other question types keep speaking the single word. Auto-speak stays disabled for sentence-cloze.

### Feasibility (researched)
All three engines speak full English sentences:
- iOS `AVSpeechSynthesizer`
- HarmonyOS `textToSpeech` (CoreSpeechKit) — handles English in the zh-CN engine transparently
- Android `android.speech.tts.TextToSpeech`

### Approach
Reconstruct the sentence at speak time: `spokenText = sentenceTemplate.replace(BLANK, answer)`.

The cloze generator blanks the **exact** answer word (case-insensitive, no stemming), so this reconstruction is grammatically correct — it differs from the original example only in possible capitalization, which TTS ignores. No data-model change is required; `sentenceTemplate` and `answer` are already on the `Question` model at speak time. `BLANK` is `____` (four underscores) on HarmonyOS; confirm the same token on iOS/Android (frozen replicas).

| Platform | Speak entry point | Change |
|---|---|---|
| HarmonyOS | `BattlePage.ets` (`speakWord(lastAnswerWord)`, ~line 768) | For SentenceCloze, compute + speak the reconstructed sentence (store a `lastSpeakText` rather than just the word). |
| iOS | `AppCoordinator.swift:402` (`speakCurrentBattleAnswer()`) | For `.sentenceCloze`, speak the reconstructed sentence; else speak `answer`. |
| Android | `CocosBattleBridge.kt:425` (`handleSpeakAnswer()`) | For SentenceCloze, speak the reconstructed sentence; else speak `correctAnswer`. |

## Item 3 — Cocos battle character aspect ratio

### Root cause
Cocos character sprite frames are **trimmed** (`trimType: auto` in `*.png.meta`). The idle magician artwork is **363×477 (tall, aspect 0.76)** padded inside a 512×512 square. `loadCharacterSprite` sizes the node from `frame.originalSize` (the padded **512×512 square**) and `Sprite.SizeMode.CUSTOM` then stretches the *trimmed* tall content to fill that square box → the character renders ~31% wider than tall (0.76 → ~1.0). Native (SwiftUI `.scaledToFit()` on the full square SVG) keeps the true tall proportions, hence the visible mismatch.

### Fix
`cocos/assets/scripts/ui/nodeFactory.ts` → `loadCharacterSprite`: size the node from the **trimmed rect** (`frame.rect`) scaled by the full-frame scale, instead of the padded square:

```ts
const fullScale = Math.min(fitWidth / orig.width, fitHeight / orig.height); // 185/512
node.getComponent(UITransform)!.setContentSize(
  frame.rect.width  * fullScale,   // 363 * 185/512 ≈ 131
  frame.rect.height * fullScale,   // 477 * 185/512 ≈ 172  → true 0.76 aspect
);
```

Properties:
- Gives the character its true aspect at the **same fraction-of-box size** as native (preserves the margin look).
- Per-frame correct: each pose (`Fight` is genuinely wide 1.13, `Beaten` near-square) renders at its own trim aspect, recomputed on every `loadCharacterSprite` / `swapTexture` call.
- Degrades to current behavior for any untrimmed frame (`rect == originalSize`).
- Horizontal centering: trimmed content is centered in the card. We do **not** replicate native's small (~10px) natural off-center offset — centered art reads cleanly. (Decided.)

Cocos-only change shared by all three embeds; no native code touched. After the edit, rebuild + re-embed Cocos per platform (`tools/cocos/build-ios.sh`, `build-harmonyos.sh`, `build-android.sh`).
