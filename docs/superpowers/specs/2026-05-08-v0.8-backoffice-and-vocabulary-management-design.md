# V0.8 Backoffice and Vocabulary Management Design

- **Date:** 2026-05-08 (product baseline); **Revision:** 2026-05-17 — V0.8.1 + V0.8.2 marked **completed**; aligned with [`docs/WordMagicGame_roadmap.md`](../../WordMagicGame_roadmap.md) and three-platform bootstrap parity (V0.7.1).
- **Status:** **V0.8.1 and V0.8.2 — completed** (server + HTML surfaces + pytest). Parent vocabulary workspace and child merged pull are live; system administrator console under `/admin/` uses session cookie auth and `admin_audit_service` for high-risk actions.
- **Roadmap:** V0.8 is re-scoped as a backend-management release split into V0.8.1 and V0.8.2 — both **shipped**.
- **Primary goal:** Make vocabulary, parent, device, and operational management reliable across tenant-scoped families.
- **Version split:** V0.8.1 = parent vocabulary management; V0.8.2 = system administrator console.

### Implementation baseline (V0.8.1 — shipped)

- **JSON APIs** extend existing legacy prefixes (`/api/v1/family/{family_id}/family-packs/*`, `/api/v1/family/{family_id}/*`). Per [`.cursor/rules/api-route-pattern.mdc`](../../../.cursor/rules/api-route-pattern.mdc), **canonical family-scoped URLs** are `/api/v1/family/{family_id}/...`; the repo’s `attach_legacy_aliases()` clones each legacy route onto that shape for backwards compatibility (aliases omitted from OpenAPI).
- **Merged child vocabulary:** **`GET`/`HEAD`** **`/api/v1/family/{family_id}/packs/latest.json`** returns global (`pack_service` / `WordPack` pointer) plus published family slices (`family_pack_service.collect_child_vocabulary`). **`GET`/`HEAD`** **`/api/v1/family/{family_id}/family-packs/latest.json`** remains **family-only** merged slices (V0.6.3 contract); clients needing global+family must call **`/api/v1/family/{family_id}/packs/latest.json`** (or the equivalent **`/api/v1/family/{family_id}/child/packs/latest.json`** alias).
- **Optional header:** `X-Family-Id` mismatch with `DeviceBinding.family_id` → **`403`** error body **`TENANT_MISMATCH`** (merged endpoint).
- **Import pipeline:** `family_pack_import_service` delegates storage + vision to **`lesson_service`** (`upload_lesson_image`, `extract_lesson_payload` JSON), then maps `words[]` into family draft rows with the `fam-<8hex>-` prefix contract.
- **Parent HTML** `/family/{family_id}/packs/*` uses **cookie soft-auth** (redirect `303` → `/family/{family_id}/login` when no session), consistent with `/family/{family_id}/` dashboard — not raw `Depends(current_parent_user)` JSON 401 for anonymous browsers.

**Primary modules:** `server/app/routers/parent_packs_pages.py`, `server/app/routers/parent_family_pack.py`, `server/app/routers/child_family_pack.py`, `server/app/services/family_pack_service.py`, `server/app/services/family_pack_import_service.py`. **Tests:** `test_parent_packs_pages.py`, `test_child_packs_merged.py`, `test_family_pack_batch.py`, `test_family_pack_import.py`. **Plan:** [`2026-05-08-v0.8.1-parent-vocabulary-management.md`](../plans/2026-05-08-v0.8.1-parent-vocabulary-management.md).

### Implementation baseline (V0.8.2 — shipped)

- **HTML console** under **`/admin/`** (separate from `/family/{family_id}/`): login/logout via **admin session cookie** (`admin_session_cookie_name`); `_require_admin_html` gates all console pages — anonymous users redirect to `/admin/login`.
- **Pages (implemented):** overview `/admin/`; `/admin/parents` + detail; `/admin/devices`; `/admin/global-packs` (+ per-pack detail, publish/rollback, image import to draft); `/admin/family-packs` (inspect, unarchive, rollback); `/admin/audit-logs`; user feedback triage where wired.
- **High-risk actions** call **`record_admin_action`** in `server/app/services/admin_audit_service.py` (reuses **`AuditLog`** / `ActorRole.ADMIN`); forms require **reason text** + confirmation per §9.
- **Services:** `server/app/services/admin_console_service.py`, `server/app/services/admin_console_overview_service.py`; global pack ops delegate to existing `global_pack_service` / `pack_service`.
- **Tests:** `server/tests/test_admin_pages.py` (login redirect, overview, parents/devices/family-packs/global-packs flows, audit log visibility).
- **Note:** Legacy **JSON** admin APIs from V0.5.x (`/api/v1/admin/words`, `/api/v1/admin/packs`, …) remain for tooling; **day-to-day operations** for system admins use the V0.8.2 HTML console, not unauthenticated JSON calls.

## 1. Background

The product already has several backend foundations:

- Parent web shell under `/family/{family_id}/`, including login, dashboard, bound devices, redemptions, inbox, and account settings.
- Family-scoped device binding through `DeviceBinding`, with device tokens resolving to a concrete `family_id`.
- Family word-pack primitives: `FamilyPackDefinition`, `FamilyPackDraft`, `FamilyWordPack`, and `FamilyPackPointer`.
- Global content backend primitives for words, categories, global word-pack snapshots, and admin publish/rollback APIs.

V0.8 delivered the coherent management surface that was missing before this release. V0.8 is not an AI story or battle-presentation release: it makes the backend usable so parents can manage their own vocabulary packs and system administrators can inspect and safely intervene in users, devices, global vocabulary, and family vocabulary.

## 2. Goals

- Treat each parent/family as a tenant and make tenant isolation explicit in API and UI design.
- Split vocabulary into shared **global vocabulary** and tenant-owned **family vocabulary**.
- Let parents create, import, edit, validate, publish, and rollback their own family word packs.
- Let child clients fetch a merged global + family vocabulary response scoped by their bound device token.
- Add a separate system administrator web console under `/admin/`.
- Allow system administrators to inspect all families and perform limited high-risk operations with audit logs.
- Preserve the existing parent web shell and family-pack API direction instead of replacing it with a new stack.

## 3. Non-Goals

- Do not implement AI story generation or personalized adventure storylines in V0.8.
- Do not move battle audio, BGM mixing, or Cocos2D battle presentation forward in V0.8.
- Do not allow parents or system administrators to bypass validation and publish AI-extracted words directly.
- Do not allow system administrators to silently edit a parent's draft words in V0.8.2.
- Do not merge global word packs and family word packs into one collection.
- Do not trust a client-supplied family or parent identifier as the authorization source.

## 4. Version Scope

### 4.1 V0.8.1 - Parent Vocabulary Management

> **Status: completed (2026-05-17).**

V0.8.1 extends the existing `/family/{family_id}/` web backend with a vocabulary workspace. It focuses on the parent's complete loop:

```text
Create pack
  -> import image or manually enter words
  -> review and edit draft table
  -> validate
  -> publish to family
  -> child devices sync merged global + family vocabulary
  -> rollback if needed
```

### 4.2 V0.8.2 - System Administrator Console

> **Status: completed (2026-05-17).**

V0.8.2 adds a separate `/admin/` web backend. It focuses on global visibility and controlled intervention:

```text
Inspect parents / families / devices / vocabulary
  -> diagnose issues
  -> perform limited high-risk actions
  -> require confirmation and reason
  -> write audit logs
```

## 5. Tenant and Vocabulary Model

### 5.1 Tenant Boundary

Each parent-owned family is a tenant. The canonical tenant key is `family_id`, backed by the `Family` model and referenced by users, devices, child profiles, wishlist items, and family word packs.

Product language may call this a "parent number" or "family number", but authorization must use server-derived `family_id`.

### 5.2 Client Pull Authentication

Child clients may send an explicit family or parent hint for observability and configuration checks, but the server must derive the real tenant from the device token:

```text
Authorization: Bearer <device_token>
optional X-Family-Id: fam-xxxxxxx

server:
  decode device token
  load DeviceBinding by binding_id
  reject revoked bindings
  use DeviceBinding.family_id as the real tenant
```

If the optional tenant hint is present and does not match the device binding's `family_id`, the server should return `403 TENANT_MISMATCH`. This makes misconfigured clients visible instead of silently serving data for a different tenant.

### 5.3 Vocabulary Types

| Type | Owner | Scope | Storage direction |
| --- | --- | --- | --- |
| Global vocabulary | System admin | Shared by all families | Existing global `Word`, `WordPack`, `PackPointer` stack |
| Family vocabulary | Parent tenant | One `family_id` only | Existing `FamilyPackDefinition`, `FamilyPackDraft`, `FamilyWordPack`, `FamilyPackPointer` stack |

Global vocabulary is the baseline. Family vocabulary is a tenant-specific overlay.

### 5.4 Merge Rules

Child clients should receive a deterministic merged response:

1. Load current global packs.
2. Load current active and published family packs for the device's `family_id`.
3. Apply global words first.
4. Apply family words second.
5. If a family word uses the same `word_id`, the family entry overrides allowed fields.
6. If a family draft entry marks a global word as hidden, the merged response excludes that global word for this family.
7. Family custom word IDs must follow the existing `fam-<family-slice>-...` prefix contract.
8. The response ETag is private and includes both global pack versions and family pack versions.

## 6. V0.8.1 Parent Pages

### 6.1 Navigation

The parent web shell should add a "词库" navigation item. The shell remains quiet and operational, similar to the existing parent dashboard.

Recommended top-level navigation (implemented strip on `_nav.html`; tight layouts hide secondary links on small screens):

```text
首页 | 词库 | 兑换 | 消息 | 设置
```

Device management stays primary from `/family/{family_id}/` dashboard and `/family/{family_id}/devices/*` routes rather than duplicating a top-nav link on narrow layouts.

### 6.2 Routes

```text
GET  /family/{family_id}/packs
GET  /family/{family_id}/packs/new
POST /family/{family_id}/packs
GET  /family/{family_id}/packs/{pack_id}/import
POST /family/{family_id}/packs/{pack_id}/publish
GET  /family/{family_id}/packs/{pack_id}/versions
GET  /family/{family_id}/packs/{pack_id}
```

Image bytes are **not** posted to a second HTML path: the import page’s form uses **`POST /api/v1/family/{family_id}/family-packs/{pack_id}/import-image`** (multipart) so the same auth and validation as the JSON API apply. Rollback remains available via the **existing** JSON route `POST /api/v1/family/{family_id}/family-packs/{pack_id}/rollback` (not duplicated as an HTML form in v1).

HTML routes use the same `family_id` scoping as JSON: unauthenticated users receive **`303`** to **`/family/{family_id}/login`**. Cross-family access continues to return non-leaking **`404`** where applicable.

### 6.3 Pack List

`/family/{family_id}/packs` shows:

- pack name and description;
- state: active or archived;
- draft word count;
- current published version;
- current published word count;
- whether unpublished changes exist;
- last updated time;
- primary actions: open, create, archive/unarchive.

### 6.4 Pack Detail and Table Editor

`/family/{family_id}/packs/{pack_id}` is the core workspace:

- metadata panel: name, description, state, current version, draft word count;
- import entry: link to `/family/{family_id}/packs/{pack_id}/import` (form posts to **`import-image`** API);
- draft table (read-only listing in v1 HTML): English word, Chinese meaning, category, difficulty — sourced from serialized draft;
- publish panel with notes (HTML form → **`POST /family/{family_id}/packs/{pack_id}/publish`** → service `publish`);
- optional rollback continues to use **`POST /api/v1/family/{family_id}/family-packs/{pack_id}/rollback`** (API or future HTML wrapper).

**Incremental scope:** in-browser row edit + batch paste UX may layer on **`htmx`** later; **v1 ships row edits via** **`POST .../draft/words:batch-upsert`** and single-word **`PUT .../draft/words/{word_id}`**. Publishing remains blocked when the draft is empty or exceeds caps (**`409`** **`EMPTY_PACK`** / **`PACK_FULL`** etc.).

### 6.5 Image Import

Image import should create or update a draft only. It must not publish.

Flow:

```text
Parent uploads textbook or workbook image
  -> server stores image if configured (`lesson_service.upload_lesson_image`)
  -> LLM/vision extracts lesson-shaped JSON (`lesson_service.extract_lesson_payload`)
  -> `family_pack_import_service` maps `words[]` into custom draft rows (`batch_upsert_draft_words`)
  -> browser returns to pack detail after multipart POST to `import-image`
  -> parent reviews, fixes (API), deletes, and adds rows
```

Low-confidence or incomplete extraction results can enter the draft, but invalid fields are marked and block publishing.

### 6.6 Versions

`/family/{family_id}/packs/{pack_id}/versions` shows published versions, word count, publish notes, and timestamps. V0.8.1 supports rollback to the previous version. Arbitrary version diff can wait.

## 7. V0.8.1 API Changes

The existing `/api/v1/family/{family_id}/family-packs` API remains the JSON contract for family-pack management. V0.8.1 adds:

```text
POST /api/v1/family/{family_id}/family-packs/{pack_id}/import-image
POST /api/v1/family/{family_id}/family-packs/{pack_id}/draft/words:batch-upsert
```

Same handlers are reachable under **`/api/v1/family/{family_id}/family-packs/...`** via legacy aliases (decorative `family_id` until path-vs-session enforcement lands).

`import-image` writes to **`FamilyPackDraft` only** (status **`201`** on success). Misconfigured LLM (**`503`** **`LLM_NOT_CONFIGURED`** / **`502`** **`LLM_CALL_FAILED`**) and upload validation (**`415`** / **`413`** / **`400`**) match structured error bodies used elsewhere in `server/`.

`batch-upsert` supports paste-table workflows:

- Response **`FamilyPackDraftWordBatchOut`**: `accepted_count`, `error_count`, `draft`, `errors[]` with `row_index`, `word_id`, `code`, `message`.
- HTTP status: **`207`** when **`error_count > 0`**; **`200`** when all rows succeed (empty error list).

The **canonical merged child** endpoint (global + family words flat list + per-pack slices):

```text
GET  /api/v1/family/{family_id}/packs/latest.json
HEAD /api/v1/family/{family_id}/packs/latest.json
```

Wire shape: **`ChildPacksMergedOut`** — `schema_version`, `family_id`, **`global_version`**, **`family_versions`** (`pack_id` → int), `merged_at`, **`words`** (merged flat list), **`packs`** (published family slices metadata). Empty merged vocabulary → **`204`**. **`ETag`** + **`Cache-Control: private, no-cache`**; **`304`** when **`If-None-Match`** matches.

The existing **`GET`/`HEAD`** **`/api/v1/family/{family_id}/family-packs/latest.json`** keeps the **family-only** **`FamilyPacksMergedOut`** contract for older clients. **HarmonyOS / iOS / Android** child sync should use **`/api/v1/family/{family_id}/packs/latest.json`** (or the **`/api/v1/family/{family_id}/child/packs/latest.json`** legacy alias) when consuming global+family together — aligned with V0.7.1 three-platform parity where applicable.

**Supporting tooling:** `scripts/seed_from_rawfile.py` resolves the rawfile at **`harmonyos/entry/src/main/resources/rawfile/data/words_v1.json`** first (V0.7 monorepo), falling back to legacy **`entry/...`** if present.

## 8. V0.8.2 System Admin Console

### 8.1 Route Boundary

System admin pages live under `/admin/`, separate from `/family/{family_id}/`.

Parent users cannot access `/admin/`. Admin users cannot accidentally enter parent-only pages unless explicitly impersonation-like support flows are designed later. V0.8.2 does not include impersonation.

### 8.2 Pages

Recommended pages:

```text
/admin/
/admin/login
/admin/parents
/admin/parents/{parent_id}
/admin/devices
/admin/global-packs
/admin/family-packs
/admin/audit-logs
```

### 8.3 System Overview

`/admin/` shows:

- registered parent count;
- family count;
- active/revoked device count;
- global word count;
- family pack count;
- publish activity;
- recent import failures;
- recent high-risk admin actions.

### 8.4 Parent Management

`/admin/parents` supports search by email, parent number, username, or `family_id`.

`/admin/parents/{parent_id}` shows:

- account status;
- email and display name;
- `family_id`;
- child profiles;
- bound devices;
- family packs and publish history;
- recent login or session information;
- relevant audit events.

Allowed first-version actions:

- disable parent account;
- restore parent account;
- trigger account data export status view;
- inspect family packs.

### 8.5 Device Management

`/admin/devices` supports search by device ID, binding ID, or `family_id`.

Allowed first-version action:

- revoke or unbind abnormal devices.

The action requires confirmation and a reason.

### 8.6 Global Vocabulary Management

`/admin/global-packs` manages global vocabulary. It can use the existing global word and pack publish/rollback models.

V0.8.2 should prioritize:

- list current global pack version;
- inspect words and categories;
- publish global pack;
- rollback global pack.

Full CMS-style authoring and content campaigns can wait.

### 8.7 Family Vocabulary Inspection

`/admin/family-packs` is primarily a support and diagnostics view:

- search family packs by family, parent, pack ID, or name;
- inspect current version, draft count, published count, and state;
- restore an accidentally archived pack;
- roll back a family pack to the previous version.

V0.8.2 does not allow system administrators to silently edit parent draft words.

## 9. Admin Audit Requirements

High-risk admin actions require:

- authenticated admin user;
- target object type and ID;
- action name;
- required reason text;
- confirmation step;
- timestamp;
- request ID when available;
- before/after state summary where practical;
- success or failure status.

Actions that must be audited:

- disable or restore parent account;
- revoke or unbind device;
- publish or rollback global pack;
- restore or rollback family pack;
- failed high-risk action attempts.

## 10. Error Handling

| Scenario | Expected behavior |
| --- | --- |
| Image import fails | Draft remains unchanged; page explains the failure and allows retry. |
| AI extracts incomplete rows | Rows appear in draft with validation errors; publish is blocked. |
| Batch paste has mixed validity | Valid rows can be saved; invalid rows return row-level errors. |
| Empty draft publish | 409 `EMPTY_PACK`. |
| Pack exceeds limit | 409 `PACK_FULL` or `WORD_LIMIT_EXCEEDED`. |
| Cross-family parent access | 404 `PACK_NOT_FOUND` or equivalent non-leaking not found response. |
| Device tenant hint mismatch | 403 `TENANT_MISMATCH`. |
| Revoked device pulls packs | 404 `BINDING_REVOKED` or existing revoked-binding error. |
| High-risk admin action without reason | 422 `REASON_REQUIRED`. |

## 11. Testing and Verification

### 11.1 Server Tests

Required coverage:

- Parent A cannot read or mutate Parent B's family packs.
- Device A can only pull packs for Device A's bound `family_id`.
- Optional tenant hint mismatch returns `TENANT_MISMATCH`.
- Global + family merge is deterministic.
- Family overrides and hidden global entries work.
- ETag changes when a global pack version changes.
- ETag changes when a family pack version changes.
- Image import writes draft only and does not publish.
- Batch upsert reports row-level errors.
- Admin high-risk actions write audit logs.

### 11.2 HTML Route Tests

Required coverage:

- `/family/{family_id}/packs` redirects anonymous users to `/family/{family_id}/login`.
- Parent pack pages only show the current family's packs.
- Pack detail renders draft count, current version, and unpublished-change state.
- `/admin/` rejects non-admin users.
- Admin high-risk forms require a reason.

### 11.3 E2E Preview Flow

Recommended preview scenario:

```text
Create parent A
  -> bind device A
  -> parent A creates a family pack
  -> parent A imports or manually enters 3 words
  -> parent A publishes
  -> device A pulls /api/v1/family/{family_id}/packs/latest.json
  -> response includes global + family words
  -> create parent B and device B
  -> parent B and device B cannot see parent A family words
  -> admin revokes device A with reason
  -> device A pull fails and audit log contains the revoke action
```

## 12. Acceptance Criteria

- [x] A parent can create a family vocabulary pack from the web backend.
- [x] A parent can import image-extracted words into an editable draft.
- [x] A parent can manually add, edit, delete, and batch paste draft words.
- [x] Invalid words block publish with clear row-level feedback.
- [x] A parent can publish and rollback a family pack.
- [x] A bound child device receives global + family vocabulary through the merged child endpoint.
- [x] Tenant isolation tests pass (`test_child_packs_merged.py`, `test_parent_packs_pages.py`, family-pack isolation tests).

### V0.8.2 — met

- [x] An admin can sign in to a separate `/admin/` console (session cookie).
- [x] An admin can search and inspect parents, families, devices, global packs, and family packs.
- [x] An admin can disable/restore parent accounts, revoke devices, rollback global packs, and restore/rollback family packs (with reason + audit).
