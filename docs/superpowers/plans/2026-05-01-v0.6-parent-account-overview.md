## TDD Discipline (per sub-version)

For each new endpoint / behaviour:

1. **RED** — write the smallest pytest / hypium case that asserts the desired behaviour. Run it → expect failure with a clear `404 not found`, `ImportError`, `AttributeError`, etc.
2. **GREEN** — write the minimum production code that satisfies the test.
3. Re-run only the new test, then the full suite, to confirm 0 regressions and 0 warnings.
4. Move to the next behaviour.

Mocking conventions (server):

- `OPENAI_API_KEY=""` autouse fixture continues; new `_disable_smtp` autouse fixture clears `SMTP_USERNAME=""` + `SMTP_PASSWORD=""` so `GmailSmtpProvider.send` no-ops with a warning. Tests that exercise the email path monkeypatch the *imported* `send_email` (or directly stub the injected `EmailProvider`) in the router module (mirrors `test_admin_llm_endpoint.py`). The provider injection point is `app.main.app.state.email_provider`, settable in tests via a `email_provider` fixture that returns an in-memory `RecordingEmailProvider`.
- Mongo continues to be `mongomock-motor` via `tests/conftest.py` document_models fixture (extend list per sub-version).
- HTML route tests use `BeautifulSoup4` (add to `pyproject.toml` test deps in V0.6.1).
- Time injection: services accept a `now: Callable[[], datetime]` factory so OTP / pair_token expiry are deterministic.

Mocking conventions (client):

- `HttpRequester` injection seam matches V0.5.
- `StringPreferencesLike` injection seam matches V0.5.
- New `AssetStoreLike` interface for DeviceIdProvider so tests can inject a Map-backed adapter.
- New `BarcodeScannerLike` interface for ScanBindingPage so UI tests can fake the scan result without bringing up the real camera UI.
- Time / nowMsProvider injected into polling services.

Linting gate after every commit:

## V0.6.1 — Walking Skeleton: Parent OTP Auth + Web Shell

**Goal:** Parent enters email → receives 6-digit OTP → enters code → Family + User created → cookie session → empty dashboard renders. Zero client changes. End-to-end Vercel deploy verifies cookie auth + Jinja templates work in production.

**New files (server):**

- `app/models/family.py`, `app/models/email_verification.py`.
- `app/schemas/parent_auth.py` — `RequestCodeIn`, `VerifyCodeIn`, `ParentMeOut`.
- `app/services/otp_service.py` — `request_code(email) -> EmailVerification`, `verify_code(email, code) -> User`. Bcrypt-hashes OTP at rest. Per-email 1/min rate limit. Per-OTP 5-attempt cap.
- `app/services/family_service.py` — `create_family_for_parent(email) -> (Family, User)` atomic.
- `app/services/email_provider.py` — defines `EmailProvider` Protocol with `async def send(*, to: str, subject: str, html: str, text: str) -> None`. Concrete `GmailSmtpProvider` constructed from `settings.smtp_*`; uses `aiosmtplib.send` with STARTTLS; on init asserts `settings.smtp_from_email == settings.smtp_username` (Gmail SPF/DKIM constraint) and fails fast at startup if mismatched. `send` no-ops with a `logger.warning` when `smtp_username=""` (test/dev). Catches `aiosmtplib.SMTPAuthenticationError` and re-raises after metric `metric_smtp_auth_fail`; catches generic `aiosmtplib.SMTPException` → logs at error and re-raises.
- `app/services/notification_service.py` — `send_otp_email(email, code, expires_in)` formats Chinese-only HTML+text body (per spec §14 r3 lock-in #1) and delegates to `app.state.email_provider.send(...)`. Catches send exceptions and translates them to a `EmailDeliveryDegraded` warning that the router surfaces as `EMAIL_DELIVERY_DEGRADED` while still persisting the `EmailVerification` row.
- `app/routers/parent_auth.py` — `/api/v1/parent/auth/{request-code,verify-code,logout}` + `/api/v1/parent/me`.
- `app/routers/parent_pages.py` — `/parent/{login,verify,/}` HTML routes; `/parent/auth/{request-code,verify-code,logout}` form-submit shims that wrap the API and return HTML / Set-Cookie / redirect.
- `app/templates/_base.html`, `_nav.html`, `parent/{login,verify,dashboard}.html`.
- `app/static/parent.css`.
- `tests/test_parent_otp.py`, `tests/test_parent_pages.py`.

**Modified files (server):**

- `app/models/user.py` — add `family_id`, `display_name`, `email`, `timezone`, `scheduled_deletion_at`; `password_hash` becomes `str | None`. Backfill: existing admin rows keep their bcrypt hash; parent rows always have None.

**Endpoints (V0.6.1):**

| Method | Path | Auth | Behaviour |
|---|---|---|---|
| POST | `/api/v1/parent/auth/request-code` | none | 202 always; rate-limited per-email 1/min; sends OTP email if rate-limit clear |
| POST | `/api/v1/parent/auth/verify-code` | none | 200 + cookie when (email, code) valid + atomic create user/family if new; 403 INVALID_CODE; 410 CODE_EXPIRED; 429 TOO_MANY_ATTEMPTS |
| POST | `/api/v1/parent/auth/logout` | cookie | 200 + clear cookie |
| GET | `/api/v1/parent/me` | cookie | 200 `{ id, email, displayName, family_id, role, timezone }`; 401 if no cookie; 403 if admin cookie |
| GET | `/parent/login` | — | HTML email-input page |
| GET | `/parent/verify?email=...` | — | HTML 6-digit OTP page |
| POST | `/parent/auth/request-code` (form) | — | Forwards to API; renders verify page on 202 |
| POST | `/parent/auth/verify-code` (form) | — | Forwards to API; sets cookie; redirects to /parent/ |
| GET | `/parent/` | cookie | HTML dashboard skeleton; redirect to /parent/login if no cookie |

**Behaviour contracts (server, ≥14):**

1. POST request-code with new email → 202; EmailVerification row pending; send_email called once.
2. POST request-code twice within 60s → second call returns 202 but send_email NOT called (rate limit).
3. POST request-code for email that already has 1 pending → invalidates old (status=expired), inserts new.
4. POST verify-code with correct code → 200 + cookie + Family + User created; verification status=used.
5. POST verify-code with bad code → 403; attempts incremented; verification still pending.
6. POST verify-code 5th bad attempt → 410 CODE_EXPIRED; verification status=expired.
7. POST verify-code after expires_at → 410; verification auto-marked expired.
8. POST verify-code for already-existing parent email → 200 + cookie; no duplicate Family / User created.
9. POST verify-code with admin's email → 403 ROLE_MISMATCH (admin must use /api/v1/auth/login).
10. GET /api/v1/parent/me without cookie → 401.
11. GET /api/v1/parent/me with admin cookie → 403.
12. GET /parent/ without cookie → 302 to /parent/login.
13. GET /parent/login HTML contains `<form action="/parent/auth/request-code">` and Tailwind/HTMX scripts.
14. GET /parent/ with cookie → contains `<div id="devices-grid">` and parent's display_name / email prefix.
15. POST logout → cookie cleared.
16. Cookie issued from > 7 days ago is renewed on next /me request (Set-Cookie present in response).

**Behaviour contracts (client):** none. Client unchanged.

**Acceptance:**

- `uv run pytest` green, 0 warnings.
- `vercel deploy --prod` succeeds; `curl https://happyword.vercel.app/parent/login` returns HTML 200 with email form.
- Manual: with `EMAIL_PROVIDER=gmail_smtp` and a real Gmail App Password set on Vercel, register a parent on prod, see the OTP email arrive (also visible in the sender Gmail's "Sent" folder), verify it, see dashboard render. Confirm at least one external recipient (e.g. a QQ/163 mailbox) does NOT route to spam — if it does, document the workaround in §V0.6.7 runbook.

## V0.6.2 — Client Scan Binding + Device + Child Profile

**Goal:** Parent web renders a QR + 6-digit short code on `/parent/devices/add`. HarmonyOS client opens ScanBindingPage which calls ScanKit; on scan, posts to `/api/v1/pair/redeem` with its persistent device_id (AssetStoreKit-backed UUID); a DeviceBinding + ChildProfile (1:1) are atomically created; client persists the device_token and shows the bound state. Parent web HTMX poll flips the row to "已绑定 [device_id end-4]".

**New files (server):**

- `app/models/{pair_token,device_binding,child_profile}.py`.
- `app/schemas/{pair,parent_child}.py`.
- `app/services/pair_service.py` — `create_pair(parent) -> PairToken`, `redeem(token_or_short, device_id, user_agent) -> (DeviceBinding, ChildProfile, device_token)`, `cancel(token) -> PairToken`, `expire_old() -> int`.
- `app/services/child_profile_service.py` — `update(child_id, payload, parent) -> ChildProfile`, `delete(child_id, parent) -> None` (soft-delete, cascade revoke binding).
- `app/routers/pair.py` — `/api/v1/pair/redeem` (no auth) + `/api/v1/parent/pair/{create,status/{token},/{token} (DELETE)}` (cookie) + `/p/{token_short}` HTML landing.
- `app/routers/parent_api.py` — devices listing / detail; child profile PUT/DELETE.
- `app/templates/parent/{devices_add,device_detail}.html`, `partials/{device_card,pair_status_row}.html`, `pair/landing.html`.
- `tests/test_pair_flow.py`, `tests/test_device_management.py`.

**New files (client):**

- `services/DeviceIdProvider.ets` — first tries `@kit.AssetStoreKit` Asset.query("wordmagic.device_id"); on miss generates 16 bytes hex and Asset.add with `IS_PERSISTENT: true`; on add failure falls back to preferences "wordmagic_device" key "device_id". Records `device_id_source` ("asset_store" / "preferences_fallback") in CloudCredentials.
