# V0.4.7 — 自定义愿望单条目 (Implementation Plan)

> Status: in progress  
> Branch: `feat/v0.4-deep-learning-and-spelling`  
> Author: assistant (autonomous)

## Goal

Let the parent add (and later remove) custom wishlist items beyond
the three baked-in defaults (`看 iPad 20 分钟`, `手表零钱充值 10 元`,
`买一个礼物 (≤20 元)`). The kid sees their family's actual reward
catalog. We **do not** introduce real payment; redeeming a custom
wish still goes through the existing PIN → CoinAccount.redeem →
GiftBox flow.

## Out of scope

- Editing an existing custom wish (delete + re-add covers the same
  user need with less UI surface).
- Real payment / mini-program integration.
- Cloud sync of custom wishes.
- Per-wish images beyond emoji (no upload pipeline today).

## Decisions

1. **Storage.** Reuse `WishlistStore`. Custom wishes live in the
   same `WishlistSnapshot.wishes` array with an `isCustom: true`
   flag. Snapshot version bumps to 2 so future migrations have a
   marker; `parseWishlistSnapshot` is forward-compatible (older v1
   blobs default `isCustom=false` per wish).
2. **Identity.** Custom wish ids use the prefix `custom-` plus a
   timestamp + a small random suffix:
   `custom-${nowMs}-${rand6}`. Catalog backfill on `init` keeps
   skipping ids it already has, so adding a custom wish then
   re-launching the app preserves it without duplication.
3. **Catalog default contract.** `WISHLIST_CATALOG` entries stay
   `isCustom=false`. The "default catalog" set is closed (3 items)
   for V0.4.7 — re-adding a default that the parent removed is
   undefined behaviour and we don't try to handle it (you can't
   remove default items in V0.4.7 anyway).
4. **Parent PIN gate.** Adding and removing a custom wish both
   require a successful `ParentPinDialog` clearance, mirroring the
   redemption flow (V0.3.9 wishlist refactor). When `parentPin` is
   unset on `GameConfig`, both flows surface the same "先在设置里
   配置家长密码" toast / hint that the redemption flow already uses
   — we reuse the existing helper.
5. **Add UI.** A small `➕` button on the WishlistPage header pushes
   a `AddCustomWishDialog`. The dialog has three inputs:
   - 名称 — TextInput (1-12 chars)
   - 金币数量 — TextInput numeric, range [5, 200]
   - 表情 — TextInput (1-2 chars; defaults to ⭐ on empty)
   Submit validates and calls `WishlistStore.addCustomWish`.
   On success the dialog dismisses; the new wish appears at the
   bottom of the list (sorted: defaults first, then custom in
   insertion order).
6. **Remove UI.** Each custom wish card gains a small `✕` button.
   Tapping `✕` opens the parent PIN gate; on success, an
   AlertDialog confirms; on confirm, `WishlistStore.removeCustomWish`
   drops the wish.
7. **Validation errors.** Surface inline below the form fields
   (TextInput placeholder + helper text). No toast for validation —
   only for "no PIN configured" / network-style errors that don't
   apply here.

- All on-device tests pass (36 → 38+).
- No-device unit tests green; CodeLinter clean.
- Adding a wish: tap `➕` → enter "看动画片" / 15 / 🎬 → confirm

  across page navigations.

  AlertDialog confirm → card disappears, persisted.
- Default wishes have no `✕` button (only custom wishes do).
- v1 snapshots load without crashing; their wishes default to
  `isCustom=false`.
